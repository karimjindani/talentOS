## 1. Diagnosis verification (why it was denied)

- Queried the live DB: `orgadmin@sbp.talentos.local` holds an `ORG_ADMIN` `TenantMembership` on tenant
  `sbp`, with the Keycloak subject linked. Data correct → not a code/data bug.
- Keycloak events for every historical login/first-login flow showed
  `redirect_uri="http://localhost:3200/api/auth/callback/keycloak"` → login always returned to the
  default (`demo`) tenant. **Confirmed root cause.**

## 2. Disproving the rejected approaches (empirical)

| Approach | Observed `redirect_uri` | Verdict |
| --- | --- | --- |
| Unpin `AUTH_URL` + `trustHost` | `http://<container-id>:3000/…` | ✗ container host |
| nginx proxy forwarding `X-Forwarded-Host` (set on the container directly) | `http://<container-id>:3000/…` | ✗ header ignored for callback |

Confirms next-auth v5 beta.25 derives the callback URL from a pinned `AUTH_URL`, not the forwarded host.
