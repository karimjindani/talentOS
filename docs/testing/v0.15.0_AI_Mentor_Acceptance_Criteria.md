# AI Mentor Acceptance Criteria — v0.15.0

**Code version:** `v0.15.0`  
**Branch:** `feat/applicant-ai-mentor-skeleton`  
**Date:** 2026-07-09  

---

## Overview

Acceptance criteria for every implemented feature in the AI Mentor. Each criterion must be satisfied before the feature is considered complete and ready for merge.

---

## AC-01: AI Mentor Page Rendering

**Feature:** Chat UI — Page Load  
**Given** an accepted applicant is logged in  
**When** they navigate to `/dashboard/mentor`  
**Then** the page renders with:
- ✅ A left sidebar containing "New Chat" and "Clear Chat" buttons
- ✅ A list of suggested question chips (9 items)
- ✅ A main chat area with a welcome message
- ✅ An input field with a Send button at the bottom
- ✅ No console errors or hydration warnings (non-blocking)

---

## AC-02: Send and Receive Messages

**Feature:** Chat — Message Exchange  
**Given** the AI Mentor page is loaded  
**When** the user types a question and clicks Send (or presses Enter)  
**Then**:
- ✅ The user message appears right-aligned in the chat area
- ✅ A loading indicator ("Generating response...") appears immediately
- ✅ The mentor response appears left-aligned after the LLM responds
- ✅ The response is rendered as Markdown (headers, bullets, code blocks)
- ✅ The input field is cleared after sending

---

## AC-03: Per-Conversation Loading State

**Feature:** Chat — Loading Isolation  
**Given** multiple conversations exist  
**When** a message is sent in conversation A  
**Then**:
- ✅ Only conversation A shows the loading indicator
- ✅ Switching to conversation B shows no loading indicator
- ✅ Switching back to conversation A preserves the loading state
- ✅ The response lands only in conversation A

---

## AC-04: New Chat

**Feature:** Conversation Management — New Chat  
**Given** the user has an active conversation with messages  
**When** the user clicks "New Chat"  
**Then**:
- ✅ A new conversation is created with a unique ID
- ✅ The new conversation shows the welcome message
- ✅ The input field is cleared
- ✅ The previous conversation remains in the history list
- ✅ The new conversation becomes the active conversation

---

## AC-05: Clear Chat

**Feature:** Conversation Management — Clear Chat  
**Given** the active conversation has messages  
**When** the user clicks "Clear Chat" and confirms  
**Then**:
- ✅ A confirmation dialog appears before deletion
- ✅ On confirm, the active conversation is deleted
- ✅ If other conversations exist, the most recent becomes active
- ✅ If no conversations remain, a new empty conversation is created

---

## AC-06: Conversation Switching

**Feature:** Conversation Management — Switch  
**Given** at least 2 conversations exist  
**When** the user clicks on a conversation in the history list  
**Then**:
- ✅ The selected conversation becomes active
- ✅ The chat area shows the selected conversation's messages
- ✅ The active conversation is highlighted in the history list
- ✅ No loading state is carried over from the previous conversation

---

## AC-07: Conversation Deletion

**Feature:** Conversation Management — Delete  
**Given** multiple conversations exist in history  
**When** the user hovers over a conversation and clicks the delete icon  
**Then**:
- ✅ The conversation is removed from the history list
- ✅ The conversation is removed from localStorage
- ✅ If the deleted conversation was active, the next one becomes active

---

## AC-08: Conversation Persistence

**Feature:** Persistence — localStorage  
**Given** the user has conversations with messages  
**When** the page is reloaded  
**Then**:
- ✅ All conversations are restored from localStorage
- ✅ Messages and timestamps are preserved
- ✅ Date objects are properly revived
- ✅ The `isLoading` flag is reset to false on reload

---

## AC-09: Auto-Scroll on Send

**Feature:** Chat — Auto-Scroll  
**Given** the chat area has enough messages to require scrolling  
**When** the user sends a new message  
**Then**:
- ✅ The view scrolls to the bottom immediately (not smooth)
- ✅ The scroll happens after the DOM updates (via rAF + setTimeout)

---

## AC-10: Scroll-to-Bottom Button

**Feature:** Chat — Scroll Button  
**Given** the chat area is scrolled up  
**When** the user scrolls away from the bottom  
**Then**:
- ✅ A floating scroll-to-bottom button appears
- ✅ Clicking the button scrolls to the bottom smoothly
- ✅ The button disappears when at the bottom

---

## AC-11: Suggested Questions

**Feature:** Chat — Quick Actions  
**Given** the AI Mentor page is loaded  
**When** the user clicks a suggested question chip  
**Then**:
- ✅ The question text is populated in the input field
- ✅ The user can edit the question before sending
- ✅ All 9 suggested questions are available

---

## AC-12: Markdown Rendering

**Feature:** Chat — Markdown  
**Given** the mentor returns a response with Markdown formatting  
**When** the response is rendered  
**Then**:
- ✅ Headers are rendered with appropriate styling
- ✅ Bullet points and numbered lists are formatted
- ✅ Code blocks have syntax highlighting (Prism)
- ✅ GFM tables are supported
- ✅ Links are clickable

---

## AC-13: Rich Card Components

**Feature:** Chat — Rich Cards  
**Given** the mentor response includes cards  
**When** the response is rendered  
**Then**:
- ✅ Task cards show title, description, due date, estimated time
- ✅ Progress cards show a percentage bar
- ✅ Timeline cards show a list of items
- ✅ Tips cards show a bulleted list
- ✅ Badge cards show label and value

---

## AC-14: API Authentication

**Feature:** Security — Auth Guard  
**Given** a user without a valid session  
**When** they call GET or POST `/api/ai/mentor`  
**Then**:
- ✅ The API returns 401 Unauthorized
- ✅ No conversation data is leaked

---

## AC-15: Prompt Validation

**Feature:** Security — Input Validation  
**Given** the user submits a prompt  
**When** the prompt is empty, whitespace-only, or exceeds 2000 characters  
**Then**:
- ✅ The API returns 400 Bad Request
- ✅ An appropriate error message is returned
- ✅ No LLM call is made

---

## AC-16: Tenant Isolation

**Feature:** Security — Tenant Isolation  
**Given** applicants from different tenants  
**When** they access the AI Mentor API  
**Then**:
- ✅ Each tenant sees only their own conversations
- ✅ Context building is scoped by tenantId
- ✅ No cross-tenant data is accessible

---

## AC-17: LLM Integration

**Feature:** LLM — GLM/ZhipuAI  
**Given** a valid `GLM_Z_API_KEY` is configured  
**When** the user sends a question that requires LLM processing  
**Then**:
- ✅ The LLM is called with the correct model, messages, and parameters
- ✅ The response is parsed and returned to the user
- ✅ The response status is `"ok"`
- ✅ The exchange is persisted to the database

---

## AC-18: LLM Response Completeness

**Feature:** LLM — No Truncation  
**Given** `LLM_MAX_TOKENS=1024`  
**When** the user asks a detailed question  
**Then**:
- ✅ The response is complete (no mid-sentence truncation)
- ✅ The response includes all expected sections (Summary, Details, Key Concepts, etc.)
- ✅ For "detailed" requests, up to 2048 tokens are allocated

---

## AC-19: LLM Timeout

**Feature:** LLM — Timeout Handling  
**Given** the LLM API takes longer than 60 seconds  
**When** the timeout fires  
**Then**:
- ✅ The request is aborted
- ✅ `LLM_TIMEOUT` error is thrown
- ✅ No retry is attempted on timeout
- ✅ Fallback to stub response with degradation note

---

## AC-20: LLM Error Fallback

**Feature:** LLM — Error Handling  
**Given** the LLM API fails (auth error, server error, network error)  
**When** the error is caught  
**Then**:
- ✅ Auth errors (401/403): no retry, immediate fallback
- ✅ Server errors (500): 1 retry, then fallback
- ✅ Rate limits (429): fallback to stub
- ✅ Fallback response has `status: "error"` and degradation note

---

## AC-21: Stub Mode (No API Key)

**Feature:** LLM — Stub Fallback  
**Given** no `GLM_Z_API_KEY` is configured  
**When** the user sends a question  
**Then**:
- ✅ `hasLLMConfig()` returns false
- ✅ Stub response is generated
- ✅ Response has `status: "stubbed"`
- ✅ Stub response includes context and knowledge if available

---

## AC-22: RBSE Classification

**Feature:** RBSE — Question Classification  
**Given** a user prompt  
**When** the RBSE classifies the question  
**Then**:
- ✅ Blocked questions return a refusal without LLM call
- ✅ Direct answer questions return a pre-built response without LLM call
- ✅ Complex questions are forwarded to the LLM

---

## AC-23: Knowledge Base Retrieval

**Feature:** Knowledge — RAG  
**Given** a user asks about a known topic (SDLC, SEM, testing, etc.)  
**When** knowledge retrieval runs  
**Then**:
- ✅ Relevant knowledge snippets are retrieved (max 2)
- ✅ Snippets are ranked by keyword match score
- ✅ Snippets are included in the system prompt for LLM context
- ✅ In stub mode, knowledge content is returned directly

---

## AC-24: Applicant Context

**Feature:** Context — Personalization  
**Given** an accepted applicant with a program, tasks, and missions  
**When** context is built  
**Then**:
- ✅ Context includes program name, progress percentage, upcoming tasks
- ✅ Context includes missions with week numbers and difficulty
- ✅ Context is scoped by tenantId and userId
- ✅ Safe fallback when no data is available

---

## AC-25: "Still Working..." UX Timer

**Feature:** UX — Long Loading Indicator  
**Given** the LLM is processing a request  
**When** the elapsed time exceeds 20 seconds  
**Then**:
- ✅ The loading message changes to "Still working... (Xs)"
- ✅ The elapsed seconds counter increments every second
- ✅ The counter resets when the response is received

---

## AC-26: Sidebar Navigation

**Feature:** Navigation  
**Given** an accepted applicant is on any dashboard page  
**When** they look at the left sidebar  
**Then**:
- ✅ "AI Mentor" appears as the 9th nav item
- ✅ Clicking it navigates to `/dashboard/mentor`
- ✅ The link shows active state when on the mentor page

---

## AC-27: Performance — Response Time

**Feature:** LLM — Performance  
**Given** a valid LLM configuration  
**When** the user sends a typical question  
**Then**:
- ✅ Response is received within 60 seconds
- ✅ System prompt is under 1500 characters
- ✅ Knowledge retrieval limited to 2 snippets
- ✅ Timeout set to 60 seconds with no retry on timeout
