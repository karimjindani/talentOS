# v0.11.3 — Keycloak Realm-Import Fix — Test Results

Code version: `v0.11.3` · Date: `2026-07-03`

Fixes a crash-looping Keycloak caused by an invalid `serviceAccountClientRoles` field on the
`talentos-provisioner` client in `keycloak/import/talentos-realm.json` (D-057). This is a Keycloak
realm-configuration fix; it is validated by a deployment / end-to-end test rather than the Vitest suite.

## Defect reproduced (before fix)

`docker logs talentos-keycloak` on startup:

```
ERROR: Failed to run import
ERROR: Unrecognized field "serviceAccountClientRoles" (class ...ClientRepresentation),
       not marked as ignorable ...
  ... RealmRepresentation["clients"]->ArrayList[4]->ClientRepresentation["serviceAccountClientRoles"]
```

Keycloak exits non-zero and restarts in a loop; `GET /realms/talentos/.well-known/openid-configuration`
returns connection-refused (`000`). Because the parse fails before the realm-exists check, the crash
occurs on **every** startup, fresh or existing.

## Fresh-import verification (after fix) — destructive, clean volume

Wiped the Keycloak database and re-imported from scratch:

```
docker compose stop keycloak keycloak-postgres
docker compose rm -f keycloak-postgres
docker volume rm talentos_keycloak-postgres-data
docker compose up -d keycloak-postgres keycloak     # start-dev --import-realm re-imports
```

## Automated suite

The v0.11.0 unit test `apps/admin/lib/keycloak-admin.test.ts` ("declares the talentos-provisioner
service-account client with manage-users") previously asserted the **invalid** `serviceAccountClientRoles`
field was present — i.e. it locked in the broken config — so the realm fix initially failed CI. The
assertion was updated to verify the **canonical** representation instead: the `talentos-provisioner`
client has `serviceAccountsEnabled`, and a `service-account-talentos-provisioner` user carries
`clientRoles["realm-management"]` including `manage-users`.

`npm run test` → **78 tests passed** (count unchanged — the assertion was updated, not added).
`npm run typecheck` → clean. `npm run lint` → clean (`--max-warnings=0`).

## New CI guard (regression prevention)

A `realm-import` job was added to `.github/workflows/ci.yml`: it boots
`quay.io/keycloak/keycloak:26.0` with `start-dev --import-realm` (in-memory H2, no DB) against
`keycloak/import/` and fails the build unless the `talentos` realm imports cleanly and the OIDC
discovery endpoint responds. This is the boot-level check that would have caught the original defect —
the unit test alone could not, since it had ossified the broken field. Validated locally via the
equivalent fresh-import boot (Keycloak starts, realm reachable, no `Failed to run import` /
`Unrecognized field` in logs); the job self-exercises on every push/PR in GitHub Actions.

## Notes

- No schema, data-model, or application-code change. The `talentos-provisioner` client (confidential,
  service account, secret) is unchanged — only the representation of its realm-management roles in the
  import file.
- After a clean Keycloak volume, the seeded Super Admin must complete first-login password + TOTP again
  (expected — Keycloak user state is re-imported from the realm file).
