# v0.2.0 Module Isolation Test Results

Code version: `v0.2.0`

Previous baseline: `v0.1.1`

Date: `2026-06-27`

## Scope

Validates the split of the single `web` container into two isolated containers — `talentos-applicant`
and `talentos-admin` — while keeping previously committed work functional (SDLC principles 1 and 3).

## Build Route Manifests (build-time isolation)

Applicant build exposed only: `/`, `/2fa/setup`, `/api/ai/mentor`, `/api/applications`, `/application`,
`/apply`, `/login`, `/signup` — no `/admin*` route present.

Admin build exposed only: `/`, `/applications`, `/applications/[id]`, `/programs`, `/settings` — no
applicant route present.

## Regression Coverage

The existing `@talentos/auth` security, tenant-isolation and application-workflow tests (9 tests) remain
the regression suite and continue to pass unchanged, confirming the packages were not affected by the
container split.

## Security Notes

- Module isolation reduces blast radius: the public applicant surface and the privileged admin surface
  now run as separate processes/containers and cannot reach each other's routes (verified by the 404
  isolation results above).
- `npm audit` reports 2 moderate findings, unchanged from prior baselines (carried forward).

## Conclusion

The applicant and administrator modules are isolated into two containers, all prior routes function,
cross-module routes are rejected with 404, and the regression suite passes. `v0.2.0` is validated.
