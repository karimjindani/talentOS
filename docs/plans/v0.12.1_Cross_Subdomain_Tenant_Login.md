# v0.12.1 — Cross-Subdomain Tenant Login (D-060)

Code version: `v0.12.1`
Previous baseline: `v0.12.0`
Status: Implemented

## Problem

A SUPER_ADMIN creates an organization (e.g. slug `sbp`, admin `orgadmin@sbp.talentos.local`). The org
admin opens their tenant admin portal at `sbp.localhost:3200`, completes the Keycloak first-login
(password change + TOTP), and is met with:

> **Access denied** — Your account is not a member of this organization.

despite the DB correctly holding an `ORG_ADMIN` `TenantMembership` for that user on the `sbp` tenant.

## Root cause

The membership guard, provisioning, and tenant resolution are all correct. The failure is a
**deployment/auth-topology gap**, not a code bug:

- `AUTH_URL` was pinned to `http://localhost:3200`, so next-auth built the OIDC `redirect_uri` as
  `http://localhost:3200/api/auth/callback/keycloak`. After authenticating on `sbp.localhost:3200`, the
  Keycloak callback returned the browser to **`localhost:3200`** — the default (`demo`) tenant — where the
  org admin has no membership → denial. (Confirmed in Keycloak `LOGIN`/`UPDATE_PASSWORD` events: every
  flow used `redirect_uri="http://localhost:3200/…"`.)
- `linkKeycloakIdentity` matches on email only (tenant-independent), so the Keycloak subject still linked
  on the wrong tenant — exactly the observed DB state (subject linked, but access denied).

### Why the "obvious" fixes don't work

- **Unpin `AUTH_URL` + `trustHost`**: next-auth v5 beta.25 derives the `redirect_uri` from its internal
  request origin, which in a container is the container id (`<hash>:3000`). Verified empirically.
- **Reverse proxy forwarding `X-Forwarded-Host`**: still produced the container-host `redirect_uri` even
  with the header set directly on the container. next-auth honors a pinned `AUTH_URL`, not the forwarded
  host, for the callback URL. The nginx approach was built and disproved.

## Approach — canonical host + base-domain shared cookie

The pattern that works with next-auth's URL model:

1. **Base domain** `APP_BASE_DOMAIN=lvh.me` (dev). `lvh.me`/`*.lvh.me` resolve to `127.0.0.1` with no
   host-file setup. Tenants are addressed at `<slug>.lvh.me:<port>`.
2. **Pin `AUTH_URL` to one canonical host per app** (`lvh.me:3200` admin, `lvh.me:3100` applicant). The
   apex has no subdomain → resolves to the default tenant, so it is a safe, stable login host and yields a
   fixed `redirect_uri`.
3. **Scope auth cookies to `.lvh.me`** (`packages/auth-web`): session, callback-url, csrf, pkce, state,
   nonce. The session set during the canonical-host callback is then readable on every tenant subdomain.
   Gated on a real multi-label `APP_BASE_DOMAIN` (skipped for `localhost`, which cannot carry a `Domain`).
4. **Return to the tenant subdomain after login**: `resolveTenantRedirect` allows the canonical origin and
   any subdomain of `APP_BASE_DOMAIN` (allow-list, not an open redirect); the admin/applicant middlewares
   pass the **absolute** tenant URL as `callbackUrl` so the subdomain survives the canonical-host hop.
5. The tenant guard (`resolveTenantAccess`, D-051) is unchanged — it still binds the shared session to the
   `Host`-resolved tenant via DB membership.

## Changes

| File | Change |
| --- | --- |
| `packages/auth-web/src/tenant-redirect.ts` (NEW) | Pure `isSameBaseDomain` + `resolveTenantRedirect` (allow-list redirect). |
| `packages/auth-web/src/auth.ts` | `baseDomainCookieConfig()` (Domain=`.base`), `redirect` callback via `resolveTenantRedirect`. |
| `packages/auth-web/src/index.ts` | Re-export `tenant-redirect`. |
| `apps/admin/middleware.ts`, `apps/applicant/middleware.ts` | `callbackUrl` = absolute tenant URL (`nextUrl.href`). |
| `apps/admin/app/layout.tsx`, `apps/applicant/components/PortalHeader.tsx` | Logout post-redirect uses the request Host, not the pinned `AUTH_URL`. |
| `docker-compose.yml` | `AUTH_URL`→canonical `lvh.me:<port>`, `APP_BASE_DOMAIN`→`lvh.me`; `HOSTNAME=0.0.0.0`/`PORT=3000` (fixes a latent Next-standalone boot crash: `getaddrinfo EAI_AGAIN <container-id>`). |
| `.env`, `.env.example` | Canonical `AUTH_URL`s + `APP_BASE_DOMAIN=lvh.me`. |
| `keycloak/import/talentos-realm.json` | `talentos-admin`/`talentos-applicant`: add `lvh.me` + `*.lvh.me` redirect URIs, web origins, post-logout URIs. |

No schema, data-model, or RBAC change.

## Test plan

- Unit: `packages/auth-web/src/tenant-redirect.test.ts` — the redirect allow-list (apex, subdomain,
  relative path, foreign host, look-alike suffix, malformed, single-host fallback).
- Integration (scripted, no human): provision a no-2FA `ORG_ADMIN` on a fresh tenant; drive the full
  Keycloak Authorization-Code login via curl and assert (a) `redirect_uri` is the canonical host,
  (b) the session cookie is `Domain=.lvh.me`, (c) the post-login redirect returns to the tenant subdomain,
  (d) the tenant page renders **HTTP 200 / admin portal** (not "Access denied").
- Regression: `npm run typecheck` + full Vitest suite.
