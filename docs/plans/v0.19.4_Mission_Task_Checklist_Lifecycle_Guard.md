# v0.19.4 Mission Task Checklist Lifecycle Guard

Code version: `v0.19.4`

Previous baseline: `v0.19.3`

Plan status: Implemented

## Summary

The mission task checklist (v0.19.0's fixed 3-step tasks) does not respect the assignment
lifecycle in the UI. An applicant viewing a finished (`PASSED`) or not-yet-accepted
(`NOT_STARTED`) assignment still gets an enabled "Mark as complete" button, and clicking it
surfaces the raw server error `Mission assignment is not accepted/active for this applicant.`
Additionally, a `PASSED` assignment whose completion rows are missing (e.g. data seeded or
migrated outside the normal submit flow, which gates on tasks 1–2) renders tasks 1–2 as
incomplete — reading as lost progress on a mission the applicant already passed. This version
derives the checklist as fully complete for `PASSED` assignments and renders a locked,
read-only checklist with a contextual explanation for every non-markable assignment status,
so the applicant can never trigger the raw error from the UI.

## Scope

- `packages/db/src/mission-tasks.ts`:
  - `buildTaskSummaries` becomes status-aware: a `PASSED` assignment reports all three tasks
    complete regardless of stored completion rows (a mission cannot pass through the normal
    flow without tasks 1–2 checked, since they gate "Submit for Review").
  - Export `MARKABLE_ASSIGNMENT_STATUSES` (`ACCEPTED`, `IN_PROGRESS`, `OVERDUE`) — the single
    source of truth shared by the mark guard and the UI.
  - Export `missionChecklistLockReason(status)` returning `null` for markable statuses and an
    applicant-facing explanation for the rest (`NOT_STARTED` → accept the mission first;
    `PENDING_EVALUATION`/`LATE_SUBMITTED` → submitted, awaiting review; `PASSED` → mission
    passed, checklist locked; `FAILED` → mission failed; `REPEAT` → superseded by a new
    attempt).
  - `unmarkMissionTaskComplete` now enforces the same markable-status guard as
    `markMissionTaskComplete` (previously any status could be unmarked): a locked checklist
    is immutable in both directions.
- `apps/applicant/app/dashboard/tasks/[assignmentId]/[taskIndex]/page.tsx`: read the
  assignment status; when the checklist is locked, render the lock reason instead of the
  toggle button (both the plain toggle and the `TutorialTaskGate` video path).
- `apps/applicant/app/dashboard/tasks/[assignmentId]/[taskIndex]/ToggleTaskComplete.tsx`:
  accept `locked`/`lockedReason` props; a locked task renders the reason text, not a form.

## Out of Scope

- Carrying stored task-completion rows forward into repeat attempts. Considered and rejected:
  `createRepeatMissionForSameWeekTx` assigns a **different** mission for the repeated week by
  design (v0.19.1), so "Review the Mission Brief" / "Study the Tutorial" genuinely restart —
  a fresh checklist is correct, and the locked-state messaging now makes that visible instead
  of confusing. No production path creates a second attempt of the same mission.
- Admin-portal views of applicant task state — unchanged.
- Changing the submit gate (`areRequiredMissionTasksComplete`) — unchanged.

## Test Scenarios

- **Scenario**: Passed mission shows a fully completed, locked checklist
  - **Actor**: Accepted applicant
  - **Preconditions**: The applicant's latest attempt for a week is `PASSED`; its stored
    completion rows may be absent (seeded data).
  - **Steps**: Open the Tasks page; open the task-1 resource page for that assignment; attempt
    to mark/unmark task 1 via the server helper.
  - **Expected result**: Tasks page shows all three tasks Completed; the task resource page
    shows the lock explanation and no toggle button; `markMissionTaskComplete` and
    `unmarkMissionTaskComplete` both reject with the lifecycle error.
  - **Automation**: `Added this iteration` — `missions` area scenario
    "Passed and unaccepted assignments lock the task checklist; active ones stay editable".

- **Scenario**: Repeat attempt starts with a fresh checklist, locked until accepted
  - **Actor**: Accepted applicant whose submission got the `REPEAT` review decision
  - **Preconditions**: A repeat created a `NOT_STARTED` attempt for an alternate mission of
    the same week.
  - **Steps**: Open the Tasks page; open the new attempt's task-1 page; attempt to mark task 1.
  - **Expected result**: The new attempt's tasks all show Pending (no carry-over from the
    failed attempt's different mission); the page explains the mission must be accepted first;
    `markMissionTaskComplete` rejects while `NOT_STARTED`.
  - **Automation**: `Added this iteration` — covered by the same `missions` scenario above.

- **Scenario**: Active assignment checklist remains fully editable
  - **Actor**: Accepted applicant with an `ACCEPTED` (or `IN_PROGRESS`/`OVERDUE`) assignment
  - **Preconditions**: Assignment accepted, deadline running.
  - **Steps**: Mark task 1 complete; unmark it again.
  - **Expected result**: Both operations succeed and persist; no lock message is shown.
  - **Automation**: `Added this iteration` — covered by the same `missions` scenario above.

- **Scenario**: Another tenant's applicant cannot mark or unmark these tasks
  - **Actor**: Applicant from a different tenant (attacker persona)
  - **Preconditions**: A valid `ACCEPTED` assignment exists in the victim tenant.
  - **Steps**: Call the mark/unmark helpers with the victim's assignment id but the other
    tenant's tenantId/applicantId.
  - **Expected result**: Both reject (assignment not found / not active for this applicant);
    no completion row is written or deleted across the tenant boundary.
  - **Automation**: `Added this iteration` — covered by the same `missions` scenario above.

## Security and Data Rules

- All checklist reads/writes remain scoped by `tenantId` + `applicantId` + assignment id;
  the new unmark guard tightens (never widens) the write path.
- The lock reasons expose only the applicant's own assignment status — no new data surface.
- No schema change; no migration.
