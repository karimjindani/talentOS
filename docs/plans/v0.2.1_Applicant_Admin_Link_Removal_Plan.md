# v0.2.1 Applicant Admin-Link Removal Plan

Code version: `v0.2.1`

Previous baseline: `v0.2.0`

Previous baseline commit: `fc0a690b71ef01cc07a3e5711d163040296f6a96`

Implementation commit: `83fe2cca6ded7cdbf25ae074b092fd14533b9bbd`

Plan status: Implemented

## Summary

`v0.2.0` isolated the applicant and administrator modules into separate containers, but the applicant
portal header still rendered an `Admin` link pointing at the admin host. That leaks the administrator
module into the public applicant surface and contradicts the isolation goal.

`v0.2.1` removes all admin navigation from the applicant application. Applicants no longer see an
`Admin` button, and the applicant container no longer carries the admin URL configuration.

## Implementation

- Remove the `Admin` link and the `NEXT_PUBLIC_ADMIN_URL` constant from
  `apps/applicant/components/PortalHeader.tsx`.
- Drop `NEXT_PUBLIC_ADMIN_URL` from the applicant service in `docker-compose.yml` and from
  `.env.example` (it is now unused).
- Keep the admin-to-applicant back-link (`NEXT_PUBLIC_APPLICANT_URL`) on the admin portal; only the
  applicant-to-admin direction is removed.
- Bump `@talentos/applicant` to `0.2.1` and update `Architecture.md` (portal layout cross-link).

## Scope

- Patch release; no schema change, no behavior change to admin or applicant routes themselves.
- Route-level isolation already enforced in `v0.2.0` (each container 404s the other's routes) is
  unchanged; this removes the remaining UI cross-link.

## Verification

- `npm run typecheck`, `npm run lint`, applicant build succeed.
- Rebuilt applicant container at `http://localhost:3100` renders no `Admin` link and no `localhost:3200`
  reference in the page.
