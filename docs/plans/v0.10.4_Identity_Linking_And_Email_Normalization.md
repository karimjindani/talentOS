# v0.10.4 — Identity Linking & Email Normalization (D-052)

Code version: `v0.10.4` · Previous: `v0.10.3` · Date: `2026-07-02`

## Problem (from the user-management audit, finding #2)

1. **The DB `User`↔Keycloak link was dead for admins.** `keycloakSubjectId` was only ever written by
   `provisionApplicantUser` on an applicant's first *apply*; admin/reviewer/super-admin rows stayed
   `NULL` forever even though the JWT callback computed the subject. The "linked on first login" comment
   (`packages/db/src/tenants.ts`) was false.
2. **Email casing was inconsistent** (`createOrganization` lowercased; `provisionApplicantUser` /
   `getUserByEmail` did not) against a case-sensitive `User.email @unique`, risking duplicate identities
   and orphaned lookups (e.g. the applicant status page shows "No application yet").

## Fix (code-only, no schema/migration)

- **Email normalization** — `packages/db/src/users.ts`: `normalizeEmail(email) = trim().toLowerCase()`
  applied on every write (`provisionApplicantUser`, and `createOrganization` in `tenants.ts`);
  `getUserByEmail` is now case-insensitive (`findFirst … mode: "insensitive"`).
- **Login-time subject backfill** — new `linkKeycloakIdentity({ email, keycloakSubjectId, name })`
  (case-insensitive, updates only when missing/changed, **never creates**), called best-effort from the
  admin guard `resolveTenantAccess` (`apps/admin/lib/tenant-guard.ts`) so every admin logs in linked.
  **Not** added to the jwt/session callbacks — `packages/auth-web/src/auth.ts` must stay edge-safe
  (imported by middleware); DB writes live only in server components.
- **Expose `email_verified` (no enforcement)** — jwt/session callbacks capture `profile.email_verified`
  onto `appToken.emailVerified` → `session.user.isEmailVerified` (named to avoid NextAuth's built-in
  `User.emailVerified: Date`). No gate — deferred until SMTP verification exists (self-registered
  applicants are currently unverified).

## Verification

- `npm run db:generate` + `typecheck` + `lint` + `test` green; suite **65 tests** (new
  `packages/db/src/users.test.ts`, 3 tests for `normalizeEmail`).
- Live DB checks (scratch, then removed): case-insensitive `getUserByEmail` resolves one row;
  `linkKeycloakIdentity` backfills a NULL subject and is a no-op on re-run; unknown emails return null and
  create no row.
- `docker compose up -d --build admin applicant` (exit 0); admin `307`, applicant `200`.

See test results: `docs/testing/v0.10.4_Identity_Linking_And_Email_Normalization_Test_Results.md`.
Decision: `D-052`. Out of scope: `email_verified` enforcement (SMTP), org-admin auto-provisioning
(`v0.3.1`), reserved slugs / `Application` unique constraint.
