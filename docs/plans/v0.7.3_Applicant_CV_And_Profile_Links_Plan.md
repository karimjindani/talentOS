# v0.7.3 — Applicant CV Upload + GitHub/LinkedIn on Apply

## Context

Applicants currently submit an application with only a **program** and a **motivation**
(`apps/applicant/app/apply/page.tsx`). The product needs each applicant to also attach a
**CV** and provide their **GitHub** and **LinkedIn** profiles so reviewers can assess them.

The v0.7.0 object-storage foundation (MinIO + `@talentos/storage` + the `StoredFile` model)
was built specifically for this "CV-on-apply" case, so this iteration is mostly **wiring an
existing foundation into the applicant flow** rather than new infrastructure.

Product decisions (confirmed with user):
- **CV is required** (PDF only, max 5 MB); **GitHub and LinkedIn URLs are optional**.
- Upload uses the **server-action proxy** model: the existing server-action apply form
  receives the file via multipart `FormData`, and the action streams it to MinIO server-side,
  then links it to the application. No client component, single submit. (Minor deviation from
  v0.7.0's "browser uploads direct" design, acceptable for one small CV — documented in D-045.)

Versioned as **patch `v0.7.3`** per user direction (staying in the 0.7.x line; v0.7.2 is already
taken by the deployment-validation baseline). Note: the feature is a new capability, which the
project's own convention would normally make a minor bump — recorded here as a patch by explicit
choice.

## Approach

### 1. Storage: add a server-side `putObject` helper
`packages/storage/src/index.ts` — add:
```ts
/** Server-side upload of bytes straight to MinIO (used when Next.js proxies a small upload). */
export function putObject({ key, body, contentType }: { key: string; body: Uint8Array | Buffer; contentType: string }) {
  return getS3Client().send(new PutObjectCommand({ Bucket: getBucket(), Key: key, Body: body, ContentType: contentType }));
}
```
`PutObjectCommand` is already imported. Reuse `buildObjectKey`, `getBucket` as the admin route does.

### 2. Data model: link CV + profile URLs to an Application
`packages/db/prisma/schema.prisma`, `Application` model — add:
- `githubUrl    String?`
- `linkedinUrl  String?`
- `cvFileId     String?  @unique`
- `cvFile       StoredFile? @relation("ApplicationCv", fields: [cvFileId], references: [id], onDelete: SetNull)`

`StoredFile` model — add back-relation: `cvForApplication Application? @relation("ApplicationCv")`.
Keeps `StoredFile` generic (FK lives on the `Application` side). Then generate a new migration
(`packages/db/prisma/migrations/2026063x_application_cv_links/`) via `prisma migrate dev`.

### 3. DB helpers
`packages/db/src/applications.ts`:
- Extend `CreateSubmittedApplicationInput` with optional `cvFileId?`, `githubUrl?`, `linkedinUrl?`
  and write them in the `application.create` data block (inside the existing `$transaction`).
- Add `cvFile: true` to the `include` of `getTenantApplication` (admin detail) and
  `listApplicantApplications` (applicant status view) so the CV surfaces.

### 4. Applicant apply form + action
`apps/applicant/package.json` — add dependency `"@talentos/storage": "0.7.0"`.
`apps/applicant/next.config.mjs` — add `"@talentos/storage"` to `transpilePackages`.

`apps/applicant/app/apply/page.tsx`:
- Add to the form: `<input type="file" name="cv" accept="application/pdf" required>`, and two
  optional text inputs `githubUrl` and `linkedinUrl` (with placeholders/`inputMode="url"`).
- In `submitApplication` (server action), after the existing motivation/program checks:
  - Read `const cv = formData.get("cv") as File`. Validate: present, `cv.type === "application/pdf"`,
    `cv.size > 0 && cv.size <= 5 * 1024 * 1024`. Throw clear errors otherwise.
  - Read + validate optional `githubUrl` / `linkedinUrl` with a small host-allowlist helper
    (must parse as `https?:` URL whose host ends in `github.com` / `linkedin.com`; reject otherwise).
    Empty → stored as `null`.
  - `const bytes = Buffer.from(await cv.arrayBuffer())`; `const key = buildObjectKey({ tenantId, category: "cv", filename: cv.name })`.
  - `await putObject({ key, body: bytes, contentType: "application/pdf" })`.
  - `const file = await createStoredFile({ tenantId, ownerUserId: applicant.id, bucket: getBucket(), storageKey: key, originalName: cv.name, contentType: "application/pdf", size: cv.size, category: "cv", actorUserId: applicant.id })` then `await markStoredFileReady(file.id, tenant.id)`.
  - Pass `cvFileId: file.id, githubUrl, linkedinUrl` into `createSubmittedApplication(...)`.
- Move the duplicate-application check (`findActiveApplication`) **before** the upload so we don't
  store an orphan object when a duplicate is rejected.

### 5. Admin display
`apps/admin/app/applications/[id]/page.tsx` — add an "Applicant profile" section showing:
- GitHub / LinkedIn as clickable links (`target="_blank" rel="noopener noreferrer"`), or "—" if absent.
- CV: a download link to `/api/files/{cvFileId}/download` (the existing tenant-scoped, auth-gated
  admin route in `apps/admin/app/api/files/[id]/download/route.ts` — no new endpoint needed) when
  `application.cvFile` is present.

### 6. Container config
`docker-compose.yml`, `applicant` service — mirror the `admin` service: add `minio` to `depends_on`
and the `S3_ENDPOINT / S3_REGION / S3_ACCESS_KEY_ID / S3_SECRET_ACCESS_KEY / S3_BUCKET /
S3_FORCE_PATH_STYLE` env block. `.env.example` already defines these vars (shared) — no change.

## Security (shift-left)
- CV restricted to `application/pdf` and ≤ 5 MB, validated **server-side** (not just the `accept` attr).
- Profile URLs are host-allowlisted to github.com / linkedin.com to prevent phishing/open-redirect
  links being stored; rendered with `rel="noopener noreferrer"`.
- Storage keys are tenant-namespaced (`buildObjectKey`); the download route is auth-gated and
  tenant-scoped (cross-tenant fileId → 404). Bucket stays private; bytes served via short-TTL presign.
- Known limitation (note in test doc): if application insert fails after `putObject`, the object is
  orphaned in MinIO. Acceptable for this iteration; no lifecycle cleanup yet (consistent with v0.7.0 scope).

## Verification
1. `docker compose up -d --build`, then `npm.cmd run db:migrate` (applies the new migration) and `npm.cmd run db:seed`.
2. `npm.cmd run test` (existing 33 tests stay green) and `npm.cmd run db:generate`.
3. Manual end-to-end: log in to applicant portal (`http://localhost:3100`), open `/apply`:
   - Submit **without** a CV → blocked with a clear error.
   - Submit a non-PDF / >5 MB → blocked.
   - Submit a valid PDF + GitHub + LinkedIn → redirects to `/application`.
   - Submit a bad GitHub host (e.g. `http://evil.com`) → blocked.
4. In admin portal (`http://localhost:3200/applications/{id}`): confirm GitHub/LinkedIn links and a
   working CV download (302 → presigned MinIO URL). Verify a different tenant's fileId → 404.
5. Confirm the object lands in the private `talentos` bucket under `tenant/{tenantId}/cv/...`
   (MinIO console `http://localhost:9001`).
