# v0.16.0 — Mission-Driven Dashboard Progress + Program Content Management (D-069)

Code version: `v0.16.0`

Decision: `D-069`

Type: minor (new capability + new admin surface). No schema change — `VideoResource`,
`ProgramTask` and `CalendarEvent` have existed since `v0.12.0`.

## Problem

Two gaps surfaced while exercising the `v0.15.0` mission-submission loop end to end:

1. **The dashboard ignored missions.** Its "Overall Progress", week bars and stat tiles were
   computed solely from `ProgramTask`/`UserTaskCompletion` (`v0.12.0`, pre-mission-engine). An
   applicant could submit a mission, have it reviewed and **accepted** — the core SEM learning
   loop — and no dashboard number moved.
2. **Program content was seed-script-only.** Video resources, weekly tasks and calendar events
   had no admin UI, no capability and no server actions; the only writer was the dev script
   `scripts/seed-dashboard.ts`. No role could add curriculum content through the product.

Decisions (confirmed with the product owner): dashboard progress is computed from **missions
only** (tasks remain a supplementary checklist); admin management covers the **full trio**
(resources + tasks + events).

## Changes

1. **`packages/auth`** — new `manageProgramContent` capability, granted to `ORG_ADMIN`
   (SUPER_ADMIN bypasses, as everywhere). HR and TECH_LEAD remain read-only consumers.
2. **`packages/db/src/program-content.ts`** (new) — tenant-scoped, transactional, audited CRUD
   for the three content models, mirroring `missions.ts` conventions: create verifies the program
   belongs to the tenant; update/delete are `updateMany`/`deleteMany` scoped by
   `{ id, tenantId }` and throw on zero rows; every write records an `AuditLog` entry
   (`resource.*`, `task.*`, `event.*`).
3. **`packages/db/src/submissions.ts`** — new `getApplicantMissionProgress(tenantId,
   applicantId, programId)`: per-week `{ totalMissions, acceptedMissions, percentage }` (weeks
   1–4 always present), overall `{ accepted, total, percentage }`, and `currentMission` — the
   first published mission (by week, then order) whose submission is not `ACCEPTED`, carrying its
   submission status. Only `ACCEPTED` counts as done.
4. **Applicant dashboard** (`apps/applicant/app/dashboard/page.tsx`) — Overall Progress and the
   Program Progress week bars now derive from mission progress; stat tiles are Overall Progress ·
   Missions Accepted · Tasks Completed · Days Remaining; a new **Current Mission** card (title,
   week, submission-status chip, link to the mission) sits beside Program Progress, with the
   task checklist kept below as supplementary content.
5. **Admin Program Content page** (`apps/admin/app/programs/[id]/content/`) — inline edit/delete
   rows plus add forms for the three content types, entirely server-component + server-action
   (matching the programs module); every action re-resolves the actor through
   `requireTenantAccess("manageProgramContent")` (D-051 membership-backed). "Manage content →"
   link added on the program detail page. Non-managers see the read-only notice.
6. **Regression runner** — `dashboard` area gains "Accepted mission submission moves
   mission-driven dashboard progress" (full draft→submit→accept loop against marked fixtures);
   `programs` area gains "Org Admin manages program content; roles without manageProgramContent
   are denied" (capability matrix + CRUD round-trip + audit assertions + cross-tenant delete
   rejection). Content fixtures cascade with the marked regression program on cleanup.

## Security

- Authorization: new capability enforced in the resolved tenant via `TenantMembership`
  (D-051 model); structural tenant scoping in every db helper as defense in depth.
- The admin-entered video URL is validated as well-formed http(s) (rejects `javascript:` etc.).
- All writes audited with actor attribution.
