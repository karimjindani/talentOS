# v0.15.0 AI Mentor Roadmap

## Summary

Deliver a context-aware AI Mentor that guides applicants throughout their internship rather than
functioning as a generic chatbot. The mentor is built in ten incremental phases — each phase is
independently shippable, tested, and documented per the SSDLC rules in [`sdlc.md`](../sdlc.md).

Current version baseline: `v0.14.2` (Applicant Portal Tenant Isolation, D-065).

Branch: `feat/applicant-ai-mentor-skeleton`.

## SSDLC Compliance

This roadmap follows the principles defined in [`sdlc.md`](../sdlc.md):

| Principle | How this roadmap complies |
|---|---|
| 0 — Document what you do | This plan lives in `docs/plans/`. Each phase updates it with status and results. |
| 1 — No regressions | Every phase runs the full test suite + regression suite before merge. |
| 2 — Architecture doc updated | `docs/Architecture.md` updated when the API layer (Phase 2) and DB model (Phase 3) land. |
| 3 — Testing per iteration | Each phase has a Test Plan section; regression suite updated as needed. |
| 6 — Data model updated | `docs/Data_Model.md` and `docs/Data_Dictionary.md` updated in Phase 3 (Chat History). |
| 7 — Security from iteration 1 | Tenant isolation, auth guard, and request validation are present from Phase 1–2. |

### Version & Documentation Control

- Plan stored as Markdown in `docs/plans/` ✅ (this file).
- Testing details recorded in `docs/testing/` per phase.
- Version baseline recorded in `docs/Version_Baseline.md` when `v0.15.0` is tagged.
- User-facing portal changes update `docs/user-guides/` (Phase 1 nav change, Phase 4 context questions).

### Source Control

- Branch: `feat/applicant-ai-mentor-skeleton` (trunk-based, short-lived).
- Commits follow Conventional Commits with `(v0.15.0, D-0NN)` trailers.
- Every phase lands via PR — no direct pushes to `main`.

## Phase 1: AI Mentor Chatbox UI ✅

**Goal:** Establish the complete frontend experience.

### Includes

- AI Mentor page at `/dashboard/mentor`
- Sidebar navigation link in `ApplicantShell`
- Chat interface with message history
- Input field and Send button
- Hardcoded mentor responses (frontend only)
- Responsive and professional UI using existing Tailwind brand classes

### Does Not Include

- Backend / API calls
- AI integration
- Database models
- Conversation persistence

### Key Changes

- `apps/applicant/components/ApplicantShell.tsx` — added `{ href: "/dashboard/mentor", label: "AI Mentor" }` to `APPLICANT_NAV_ITEMS`.
- `apps/applicant/app/dashboard/mentor/page.tsx` — new client component with message state, input, send button, hardcoded dummy response.
- `apps/applicant/components/ApplicantShell.test.ts` — updated nav item count (8→9), added mentor route assertions.

### Test Plan

- Unit: `ApplicantShell.test.ts` — 12 tests pass (nav item count, route matching, completeness).
- Typecheck: `npx tsc --noEmit -p apps/applicant/tsconfig.json` — passes.
- Manual: `http://demo.lvh.me:3100/dashboard/mentor` — page loads, chat UI renders, send produces dummy response.
- Docker: `docker compose build applicant && docker compose up -d applicant` — route served on port 3100.

### Status

✅ **Complete** — all tests pass, typecheck clean, Docker image rebuilt and verified.

## Phase 2: AI Mentor API Stub ✅

**Goal:** Prepare the architecture for AI integration by moving dummy responses to a backend API layer.

### Includes

- AI Mentor API endpoint (`/api/ai/mentor`)
- Chat requests sent to backend via `fetch`
- Structured dummy responses (text + rich cards)
- Request validation (non-empty prompt, max length)
- Error handling (400 for invalid input, 500 for server errors)
- Loading state in the frontend

### Does Not Include

- Real AI / LLM calls
- Database persistence
- Conversation history

### Key Changes

- `apps/applicant/lib/ai.ts` — enhance `AIInteractionResponse` with `MentorCard[]` type; keyword-based structured dummy responses (task, progress, timeline, tips, badge cards).
- `apps/applicant/app/api/ai/mentor/route.ts` — add request validation (prompt required, max 2000 chars), error handling, structured JSON response.
- `apps/applicant/app/dashboard/mentor/page.tsx` — replace `setTimeout` stub with `fetch("/api/ai/mentor")`, handle loading and error states, render rich cards.

### Test Plan

- Unit: API route validation tests (empty prompt → 400, valid prompt → 200 with structured response).
- Unit: `lib/ai.ts` keyword matching returns correct card types.
- Unit: Frontend loading state and error boundary.
- Typecheck: `npx tsc --noEmit -p apps/applicant/tsconfig.json`.
- Manual: Send messages containing "task", "progress", "timeline", "tips" — verify correct card rendering.

### Status

✅ **Complete** — typecheck clean, 12/12 nav tests pass, API stub with structured responses, validation, and error handling implemented.

## Phase 3: Chat History ✅

**Goal:** Make conversations persistent.

### Includes

- Database model (`MentorConversation`, `MentorMessage`)
- Save conversations after each exchange
- Load previous chats on page visit
- User-specific history (scoped to `userId`)
- Tenant isolation (scoped to `tenantId`)
- Conversation timestamps

### Does Not Include

- Real AI
- Mission context

### Key Changes

- `packages/db/prisma/schema.prisma` — add `MentorConversation` and `MentorMessage` models.
- `packages/db/prisma/migrations/` — new migration.
- `docs/Data_Model.md` — update with new models.
- `docs/Data_Dictionary.md` — update with field definitions.
- `apps/applicant/app/api/ai/mentor/route.ts` — persist messages to DB.
- `apps/applicant/app/dashboard/mentor/page.tsx` — load conversation history on mount.

### Test Plan

- Unit: Conversation save/load round-trip.
- Unit: Tenant isolation — user A cannot read user B's conversations.
- Unit: Messages ordered by timestamp.
- Regression: `regression:mentor` scenario — chat history survives page reload.
- Required: `npm run db:generate`, `npm run test`, `npm run build`.

### Status

✅ **Complete** — Prisma models added, migration applied, API persists and loads conversation history, typecheck clean, 12/12 nav tests pass.

## Assumptions

- Phase 1 is complete and merged on branch `feat/applicant-ai-mentor-skeleton`.
- Each phase is a separate PR; no phase is merged without passing CI and review.
- Database schema changes (Phase 3+) require migration + data model doc updates per SSDLC.
- LLM integration (Phase 6+) requires an API key configured via environment variable; no keys are committed.
- The AI Mentor is applicant-portal only; admin analytics (Phase 8) is in the admin portal.
- All phases respect existing tenant isolation and auth guards.
