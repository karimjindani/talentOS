# v0.10.0 — Super Admin Organizations Console (Tenant Creation) — Implementation Plan

Code version: `v0.10.0`

## Objective

Give the platform SUPER_ADMIN a way to create tenants from the app. Previously tenants existed only via
the DB seed script; the `createOrganization` capability was defined but wired to nothing. This slice
adds a `/organizations` console (list + create) and assigns each new tenant's first ORG_ADMIN by email.

## Why it was needed

Tenant is resolved from the URL host (`resolveTenantFromHost`), never from the user, so a super admin
always operates inside a host-resolved tenant and a slug only works once its `Tenant` row exists. There
was no create path in the product. See `Decision_Log.md` D-048.

## Decisions

- Build a UI console (not manual/CLI).
- Org admin assigned by email (DB `User` + `TenantMembership(ORG_ADMIN)`); no Keycloak Admin API.
- No schema change; new audit action `organization.created`.

## Security controls

- `can("createOrganization")` gate in **both** page and action (SUPER_ADMIN only).
- `isValidTenantSlug` server-side before insert (slug becomes a subdomain).
- Hex color validation `^#[0-9a-fA-F]{6}$`; basic email validation.
- Duplicate slug → P2002 friendly error, transaction aborts atomically.
- Every create writes an `organization.created` audit row.

## Known limitation

The assigned org admin gains real ORG_ADMIN access to their tenant only once their Keycloak account
holds the `ORG_ADMIN` realm role (DB membership provides scoping; role identity comes from Keycloak).
Full automation is the future Keycloak-Admin-API slice (D-035).
