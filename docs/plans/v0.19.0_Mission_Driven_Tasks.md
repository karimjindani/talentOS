# v0.19.0 Mission-Driven Tasks & Submissions Admin Tab

Code version: `v0.19.0`

Previous baseline: `v0.18.5`

Plan status: Implemented

## Summary

The applicant Tasks tab and mission dashboard previously showed the legacy program-level
`ProgramTask`/`VideoResource` content, disconnected from the actual mission a participant was
working on. This version replaces that with a fixed 3-task template driven directly by each
mission assignment (Review Brief → Study Tutorial → Build & Submit Evidence), gates mission
submission on completing the first two tasks, and adds a YouTube watch-gate so a tutorial task
cannot be marked complete without watching the linked video to the end. It also gives reviewers a
dedicated cross-mission **Submissions** tab in the Admin Portal instead of having to open each
mission individually to find its submissions.

## Scope

- `Mission.tutorialUrl` (optional) — an admin-authored tutorial video link per mission.
- New `MissionTaskCompletion` model tracking completion of Task 1 (Review the Mission Brief) and
  Task 2 (Study the Tutorial) per assignment attempt; unique on
  `[missionAssignmentId, taskIndex]`. Task 3 (Build & Submit Evidence) has no completion row of its
  own — it is implied complete once the linked `Submission.status` moves beyond
  `DRAFT`/`NEEDS_REVISION`.
- `packages/db/src/mission-tasks.ts`: `getMissionTasksForAssignment`,
  `listAssignedMissionsWithTasks`, `areRequiredMissionTasksComplete`, `markMissionTaskComplete`,
  `unmarkMissionTaskComplete`.
- Submission is gated on tasks 1 and 2 being complete — `saveSubmissionDraft`/`submitSubmission`
  reject a submit attempt until `areRequiredMissionTasksComplete` is true; the Submit button on the
  mission page is disabled client-side to match.
- Per-task resource page (`/dashboard/tasks/[assignmentId]/[taskIndex]`) showing the task's
  tutorial/details; Task 2 embeds a YouTube IFrame Player and only enables "Mark as complete" after
  the video reaches `YT.PlayerState.ENDED`. A mission with no `tutorialUrl` has no watch-gate — Task
  2 can be marked complete directly.
- Dashboard, Tasks and Missions pages are rewritten around `listAssignedMissionsWithTasks` instead
  of the legacy `ProgramTask`/`VideoResource` data.
- New admin **Submissions** page (`/submissions`), added to the sidebar between Missions and
  Operations, listing/filtering submissions across all missions by status and program — reuses the
  existing `reviewSubmissions` capability and the existing per-submission review page
  (`/missions/[id]/submissions/[submissionId]`); no new authorization surface.
- `packages/db/src/submissions.ts`: new `listTenantSubmissions(tenantId, { status?, programId? })`.
- Legacy `ProgramTask`/`VideoResource`/`UserTaskCompletion` tables are kept in the schema (per
  explicit product decision — "leave them unused for now") but removed from the applicant Tasks/
  Resources UI and from the admin Program Content authoring page, which now only manages Calendar
  Events.
- Security fix: `mission.tutorialUrl` is validated server-side to an http/https scheme
  (`parseOptionalHttpUrl` in the admin mission form action) and defensively re-checked before
  rendering as a link on the applicant task page, closing a `javascript:` URI XSS vector found by
  automated review during this iteration.
- Migration `20260714110000_mission_tasks`.

## Out of Scope

- Admin-authored per-mission task lists — the 3-task template is fixed, not editable per mission.
- Any completion/gating logic for Task 3 beyond the existing submission status — it deliberately
  has no independent completion row.
- Deleting the legacy `ProgramTask`/`VideoResource`/`UserTaskCompletion` tables or the admin
  program-content authoring UI for them — explicitly deferred by product decision.
- Video providers other than YouTube for the watch-gate.

## Test Scenarios

- **Scenario**: Submission is blocked until Tasks 1 and 2 are complete.
  **Actor**: accepted applicant working an accepted mission assignment.
  **Preconditions**: assignment `ACCEPTED`/`IN_PROGRESS`; Tasks 1 and 2 not yet marked complete.
  **Steps**: applicant attempts to submit evidence before completing Tasks 1/2, then marks both
  complete and submits again.
  **Expected result**: the first submit attempt is rejected server-side (and the Submit button is
  disabled client-side); after both tasks are marked complete, submission proceeds normally.
  **Automation**: Deferred — covered at unit level only (`packages/db/src/mission-tasks.test.ts`,
  `packages/db/src/submissions.test.ts`); no `scripts/regression/run.ts` scenario drives the gate
  through the real mission page yet.

- **Scenario**: Task 2 requires watching the tutorial video to completion.
  **Actor**: accepted applicant.
  **Preconditions**: the assigned mission has a `tutorialUrl` set.
  **Steps**: applicant opens the Task 2 resource page and attempts to mark it complete before the
  embedded video reaches the end, then again after watching to the end.
  **Expected result**: "Mark as complete" is disabled/rejected before `YT.PlayerState.ENDED` fires,
  and succeeds once it does.
  **Automation**: Deferred — covered at unit level only (`apps/applicant/lib/youtube.test.ts`, 9
  tests for `parseYouTubeVideoId`); the IFrame Player `onStateChange` gate itself has no Playwright
  coverage.

- **Scenario**: Task 2 has no watch-gate when the mission has no tutorial video.
  **Actor**: accepted applicant.
  **Preconditions**: the assigned mission's `tutorialUrl` is null.
  **Steps**: applicant opens the Task 2 resource page and marks it complete.
  **Expected result**: completion succeeds immediately — no video, no gate.
  **Automation**: Deferred — covered at unit level only (`mission-tasks.test.ts`).

- **Scenario (authorization)**: Only reviewers can see the admin Submissions tab's content.
  **Actor**: ORG_ADMIN/TECH_LEAD (allowed), HR (read-only), APPLICANT (denied).
  **Preconditions**: tenant has submissions in various statuses.
  **Steps**: each role loads `/submissions` and attempts to open a submission for review.
  **Expected result**: the same `reviewSubmissions` capability boundary already enforced on the
  per-mission submission review page applies — no new authorization surface was introduced.
  **Automation**: Already automated — the existing "Only Org Admin and Tech Lead can review
  submissions" `missions`-area scenario exercises the underlying `reviewSubmissions` capability that
  the new tab reuses unchanged; the `/submissions` list page itself has no dedicated scenario.

- **Scenario (negative/edge)**: `tutorialUrl` rejects a non-http(s) scheme.
  **Actor**: ORG_ADMIN authoring a mission.
  **Preconditions**: none.
  **Steps**: admin submits a mission form with `tutorialUrl` set to a `javascript:` URI.
  **Expected result**: the write is rejected server-side (`parseOptionalHttpUrl`); even if a
  malformed value existed already, the applicant task page's defensive scheme check refuses to
  render it as a link.
  **Automation**: Deferred — covered by manual security review during this iteration; no automated
  regression test asserts the rejection.

## Security and Data Rules

- `tutorialUrl` is validated to `http`/`https` on write (admin action) and re-checked on read
  (applicant task page) before being rendered as an `<a href>` — defense in depth against a
  `javascript:` URI XSS vector.
- `MissionTaskCompletion` writes are scoped by `{ tenantId, missionAssignmentId }` — an applicant
  can only mark tasks complete on their own assignment.
- The Submissions admin page performs no new authorization — it is gated by the pre-existing
  `reviewSubmissions` capability check on the underlying review action/page.
