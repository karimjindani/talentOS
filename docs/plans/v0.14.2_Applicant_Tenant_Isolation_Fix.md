# v0.14.2 — Applicant Portal Tenant Isolation Fix (D-065)

Code version: `v0.14.2` · Previous: `v0.14.1` · Date: `2026-07-05`

## Problem

The tenant-isolation fix from `v0.10.3` (D-051) was applied only to the **admin** portal. The
**applicant** portal never received the equivalent guard, so it still operated on the pre-D-051 model:
the Keycloak realm role gated portal entry and the tenant came from the `Host` header, but nothing
checked that the signed-in user is actually a member of the tenant resolved from the subdomain.

Because sessions are intentionally shared across subdomains (`Domain=.lvh.me`, `v0.12.1`/D-060) so a user
signs in once and is carried to their tenant subdomain, an authenticated user of tenant A could open
`tenantB.<host>:3100` and reach tenant B's applicant portal:

- `/dashboard` and `/application` rendered tenant B's portal (data was B-scoped, so empty for the
  intruder, but they were wrongly *present* in a tenant they do not belong to).
- Worse, `/apply`'s server action called `provisionApplicantUser({ tenantId: B.id })`, which upserts an
  `APPLICANT` `TenantMembership` — an intruder could silently enroll into tenant B.

A second, latent defect was found while verifying: the **live Keycloak realm had
`registrationAllowed: false`** (drifted from the committed import, which sets it `true`), so the
"Create account" action led to a dead login form — applicant self-registration was broken.

## Fix — port the D-051 guard to the applicant portal

Two-layer model, identical to admin: the Keycloak realm role remains the coarse **portal-entry** gate
(middleware); the DB `TenantMembership` is the **authoritative per-tenant** gate. `SUPER_ADMIN` bypasses.

- **New guard** — `apps/applicant/lib/tenant-guard.ts`, a near-exact mirror of
  `apps/admin/lib/tenant-guard.ts`:
  - `resolveTenantAccess(capability?)` — discriminated result (`ok` / `unauthenticated` /
    `unknown-tenant` / `forbidden`); reuses `getTenantContext` (`@talentos/ui`), `getTenantBySlug`,
    `getActorTenantRoles`, `linkKeycloakIdentity` (`@talentos/db`), and `isSuperAdmin` /
    `tenantRolesGrant` (`@talentos/auth`).
  - `requireTenantAccess(capability?)` — redirects on failure: unauthenticated → `/login`,
    unknown-tenant / forbidden → `/access-denied`.
- **New page** — `apps/applicant/app/access-denied/page.tsx`, a standalone page under the *root* layout
  (not the dashboard shell, so redirecting to it cannot loop). Offers "Apply to join" (`/apply`) and a
  Keycloak single-logout "Sign out" (reusing `buildEndSessionUrl`).
- **Wiring**
  - `/dashboard` (`app/dashboard/layout.tsx`) and `/application` (`app/application/page.tsx`) now call
    `requireTenantAccess("accessApplicantPortal")` first; the pre-existing accepted-only
    `/dashboard`↔`/application` business-rule redirects are preserved.
  - `/apply` (`app/apply/page.tsx`) stays **open** (public recruitment funnel — a non-member may still
    apply and thereby legitimately gain membership), but an existing member is redirected to
    `/application` via `resolveTenantAccess()`.
- **Realm config alignment** — the committed import already sets `registrationAllowed` /
  `registrationEmailAsUsername` to `true`; a new assertion in `keycloak-admin.test.ts` pins this so a
  fresh realm keeps self-registration working. (The drifted live realm was re-enabled out of band.)
- **Auth-config cleanup shipped in the same baseline** — `provisionOrgAdmin` no longer sets
  `CONFIGURE_TOTP` (2FA setup removed per operator request; avoids the Keycloak TOTP "/ by zero" error);
  the provisioner service account gains `manage-realm`/`view-users` in the import.

## Scope

Code + docs; **no schema/migration**, no data-model change, no new capability (reuses the existing
`accessApplicantPortal`). Out of scope: browser-level Playwright coverage for admin cross-tenant file /
settings denial (still tracked "Missing" in `Regression_Scenarios.md`); a permanent realm-drift remedy
(a realm reset from the committed import — deferred to avoid wiping local demo data).

See test results: `docs/testing/v0.14.2_Applicant_Tenant_Isolation_Fix_Test_Results.md`. Decision: `D-065`.
