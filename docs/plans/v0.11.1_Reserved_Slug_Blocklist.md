# v0.11.1 — Reserved Tenant-Slug Blocklist

Code version: `v0.11.1` · Previous: `v0.11.0` · Date: `2026-07-02`

## Problem (audit finding #4, remaining half)

`isValidTenantSlug` accepted routing/infra-sensitive labels (`www`, `admin`, `api`, `auth`, `keycloak`,
`minio`, `demo` — the default catch-all tenant, …). Because a tenant slug becomes a subdomain, a
SUPER_ADMIN could mint a tenant that collides with platform hosts (routing confusion / cookie-scope /
takeover). (The other half of #4 — the duplicate-active-application DB guard — already shipped via PR #13.)

## Fix (validation-only, no schema change)

- `packages/auth/src/tenant.ts`: add a `RESERVED_SLUGS` set (infra/routing + product-reserved labels);
  `isValidTenantSlug` returns `false` for a reserved slug in addition to the DNS-safe regex. Pure function,
  reused by `createOrganizationAction`; existing tenants are unaffected (validation runs only on create).
- `apps/admin/app/organizations/actions.ts`: the slug-error message now mentions reserved names.
- `packages/auth/src/auth.test.ts`: reserved slugs (`www`/`admin`/`api`/`demo`/`keycloak`) → `false`;
  normal slugs and reserved-substring slugs (`acmeapi`, `admins`, `my-demo`) still valid.

## Verification

- `db:generate` + `typecheck` + `lint` + `test` green; suite **78 tests**.
- Applied PR #13's pending migration to the local DB (`prisma migrate deploy`, after clearing a stray
  duplicate `ACCEPTED` row left over from earlier manual testing — the seed itself is idempotent and does
  not produce duplicates); confirmed the partial unique index
  `applications_applicantId_programId_active_key` exists.
- Sanity: creating an org with slug `admin`/`demo` is rejected; a normal slug succeeds.

See `docs/testing/v0.11.1_Reserved_Slug_Blocklist_Test_Results.md`. Decision: `D-054`.

## Out of scope
Custom user-management UI; SMTP/email verification. This closes the four audit findings (#1 tenant
isolation, #2 identity linking, #3 org-admin provisioning, #4 reserved slugs + duplicate-application guard).
