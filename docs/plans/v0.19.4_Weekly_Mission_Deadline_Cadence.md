# v0.19.4 Weekly Mission Deadline Cadence

Code version: `v0.19.4`

Previous baseline: `v0.19.3` (merges after `v0.19.5`, which was allocated on `origin/main` in
parallel; both ship on PR #49's branch by owner instruction — see D-091)

Plan status: Implemented

## Summary

Mission submission deadlines were a rolling offset — `deadlineAt = acceptedAt +
mission.deadlineHours` (default 168h) with `graceEndsAt = deadlineAt + gracePeriodHours`
(default 24h) — so every applicant's cutoff fell at a different weekday and hour. This change
replaces the offset with a fixed weekly cadence: missions run Monday → Thursday with an
end-of-Thursday cutoff (Thursday 23:59:59.999, server-local time), and every applicant is
guaranteed at least 4 inclusive calendar days. Accepting on Monday keeps that week's Thursday;
accepting Tuesday–Thursday rolls to the next week's Thursday; Friday–Sunday acceptances get the
coming Thursday (7/6/5 days). The 24h grace period after the cutoff is unchanged, as is every
consumer of the stored timestamps (deadline sweep, submission gating, countdown UI).

## Scope

- New pure helper `computeWeeklyDeadline(acceptedAt)` in `packages/db/src/deadline-cadence.ts`
  (exported from `@talentos/db`): earliest end-of-Thursday (server-local) whose inclusive
  calendar-day count from the acceptance date is >= 4.
- `acceptMissionAssignmentTx` (`packages/db/src/mission-assignments.ts`) computes `deadlineAt`
  via the helper and no longer reads `mission.deadlineHours`; `graceEndsAt` remains
  `deadlineAt + gracePeriodHours`.
- `Mission.deadlineHours` is retained in the schema but unused at runtime (comment updated —
  no migration, no column drop).
- New rule-based `missions` regression scenario asserting Thursday/end-of-day/>= 4 days/+24h
  grace on a real acceptance.
- Documentation: Data_Dictionary/Data_Model/Architecture notes for the unused column and new
  rule; Deployment note to pin the `TZ` env var consistently across app processes; applicant
  user-guide deadline section; Decision Log `D-091`.

## Out of Scope

- Recomputing in-flight assignments: anything accepted before this change keeps its stored
  `deadlineAt`/`graceEndsAt` (user decision — no migration, no surprises mid-mission).
- Removing the `deadlineHours` column or admin cadence configuration (none existed for the
  offset either).
- Changing the sweep (`sweepMissionDeadlines`), `submitSubmission` gating, `LATE_SUBMITTED`
  semantics, or the countdown/Days Remaining UI — all read the stored timestamps unchanged.
- Timezone configurability: the cadence anchors to the server's local timezone by decision;
  multi-region tenant-local cutoffs would be a separate iteration.

## Test Scenarios

- **Scenario**: Monday acceptance is due this Thursday
  - **Actor**: Accepted applicant
  - **Preconditions**: A `NOT_STARTED` assignment; server clock on a Monday
  - **Steps**: Accept the mission
  - **Expected result**: `deadlineAt` is that same week's Thursday 23:59:59.999 (server-local),
    `graceEndsAt` is 24h later; countdown and Days Remaining reflect it
  - **Automation**: `Added this iteration` — unit tests (`deadline-cadence.test.ts` Monday cases,
    `mission-assignments.test.ts` accept test) plus the rule-based runner scenario "Accepted
    assignments follow the weekly Thursday deadline cadence with a 24h grace period" (weekday
    coverage is deterministic in unit tests; the runner asserts the invariants on the real clock)

- **Scenario**: Mid-week acceptance rolls to next Thursday (min 4 days)
  - **Actor**: Accepted applicant
  - **Preconditions**: A `NOT_STARTED` assignment; server clock on Tue/Wed/Thu
  - **Steps**: Accept the mission
  - **Expected result**: `deadlineAt` is NEXT week's Thursday end-of-day — never fewer than 4
    inclusive calendar days (the user's "join Wednesday → next Thursday" case)
  - **Automation**: `Added this iteration` — same unit + runner coverage as above

- **Scenario**: Late submission during the post-Thursday grace period
  - **Actor**: Accepted applicant past the Thursday cutoff but inside grace
  - **Preconditions**: Assignment accepted with the cadence deadline; deadline passed < 24h ago
  - **Steps**: Submit the mission
  - **Expected result**: Submission is accepted and marked `LATE_SUBMITTED` (existing v0.18.5
    behavior, driven by the stored timestamps — unchanged by this iteration)
  - **Automation**: `Deferred` — already covered by existing v0.18.5 unit tests in
    `submissions.test.ts` (late-submission path); no new scenario-runner case because driving
    the real clock past a Thursday cutoff inside the suite would require time control the
    runner doesn't have. Recorded in Regression_Scenarios notes rather than Known Gaps since
    unit coverage exists and the cadence scenario pins the stored timestamps.

- **Scenario**: In-flight assignment keeps its pre-cadence deadline (negative/migration case)
  - **Actor**: Applicant who accepted before this change
  - **Preconditions**: Assignment with a stored offset-based `deadlineAt`
  - **Steps**: Load dashboard/missions pages; run the sweep
  - **Expected result**: The stored deadline is used untouched — no recompute
  - **Automation**: `Deferred` — nothing recomputes by construction (only
    `acceptMissionAssignmentTx` writes these fields, and only on NOT_STARTED → ACCEPTED);
    asserted implicitly by the demo tenant's existing data continuing to render. Noted in
    Regression_Scenarios notes.

## Security and Data Rules

- No new inputs, capabilities, or data surfaces; the accept path keeps its tenant + applicant +
  NOT_STARTED guards.
- No schema change; `deadlineHours` retained to avoid destructive migration.
- Cadence depends on the server's local timezone: `docs/Deployment.md` now instructs pinning
  `TZ` consistently across app processes so "Thursday" is the same on every instance.

## Tests

- New `packages/db/src/deadline-cadence.test.ts`: 7 tests (all weekdays, Monday-midnight and
  Thursday-23:59:59.999 boundaries, end-of-Thursday + >= 4-day invariant, input immutability).
- `packages/db/src/mission-assignments.test.ts`: the two offset-formula acceptance tests are
  replaced with cadence tests (Monday → this Thursday + 12h grace; Thursday → next Thursday +
  24h grace), built with local-time constructors so they pass in any timezone. 16 tests in file.
- New runner scenario (rule-based, real clock): Thursday local day, 23:59:59.999 local time,
  >= 4 inclusive days, grace exactly +24h.
