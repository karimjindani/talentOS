# v0.19.3 AI Mentor RBSE Name Blocking And Token Tracking

Code version: `v0.19.3`

Previous baseline: `v0.19.2`

Plan status: Implemented

## Summary

The AI Mentor's Rule-Based System Engine (RBSE) was allowing personal-name questions
(e.g. "explain hitesh") to reach the GLM LLM because "explain" is an allowed topic.
This version adds regex-based personal-name pattern matching at the RBSE layer so
questions about a person by name are blocked before any LLM call. Additionally, the
GLM streaming request was missing `stream_options.include_usage`, so token usage
was never returned in the SSE stream — the fix adds this field so logs now show
actual token counts instead of `?`. Finally, the Vitest config was switched from
the default `threads` pool to `forks` pool to eliminate cross-file mock
contamination that caused 6 spurious test failures under the full suite.

## Scope

- Add `PERSONAL_NAME_PATTERNS` regex array to `ai-rbse.ts` catching "explain \<Name\>",
  "who is \<Name\>", "tell me about \<Name\>", "describe \<Name\>", and
  "what do you know about \<Name\>" patterns.
- Add `NAME_PATTERN_ALLOWLIST` so technical terms (SDLC, testing, deployment, etc.)
  still pass through name-like patterns.
- Add personal identity topics to `BLOCKED_TOPICS` ("who is", "my name", "your name", etc.).
- Add `stream_options: { include_usage: true }` to GLM chat completion request so
  token counts are returned in SSE streaming responses.
- RBSE now always runs regardless of conversation history — blocked questions are
  never sent to GLM even in multi-turn conversations.
- Switch Vitest pool from `threads` to `forks` with `testTimeout: 15_000` for
  reliable test isolation under the full 43-file suite.

## Out of Scope

- Content filtering or PII detection beyond name-pattern matching — future work.
- Rate limiting or per-user token budget tracking — the token count is logged
  but no enforcement is applied.
- RBSE pattern learning or dynamic rule updates — patterns are static in code.
