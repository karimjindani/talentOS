# v0.18.2 Regression And Documentation Completeness

Code version: `v0.18.2`

Previous baseline: `v0.18.1`

Plan status: Implemented

## Summary

Closes the specific regression-coverage and documentation gaps found in manual PR review of the
`engineering-journal-mvp` branch (Engineering Journal `v0.17.0`/`v0.17.1`, Mission Assignment
`v0.18.0`): production-quality scenario coverage was missing for both features, and several docs had
drifted from shipped behavior. This is the first plan written using the new `docs/plans/TEMPLATE.md`
(`v0.18.1`, D-076) against real feature work rather than as a self-check.

## Scope

- Add a `journal` regression area (`scripts/regression/run.ts`, `RegressionArea` type, `apps/ops`
  command list/UI, `npm run regression:journal`) with real scenario coverage.
- Add `EngineeringJournalEntry` to the `RegressionDataMarker` cleanup entity types.
- Add two `missions`-area scenarios: assigned-mission-only visibility/detail/submission-drafting
  scoping, and a scenario documenting the known backfill gap for pre-existing accepted applicants.
- Update `docs/Regression_Scenarios.md` and `docs/Testing_Strategy.md` to reflect the new coverage.
- Review all project documentation for staleness against the `engineering-journal-mvp` branch's shipped
  code and correct what no longer matches (Deployment, vision, Product Backlog, user guides,
  CI/CD pipeline, Architecture, README).

## Out of Scope

- Fixing the mission-assignment backfill gap itself (add a migration/backfill script or a lazy
  on-read fallback) — this is a product decision, not a testing/documentation task; the new regression
  scenario documents and locks in current behavior so the decision isn't made by accident later.
- Automating the other pre-existing Known Gaps unrelated to this branch (storage upload/download,
  cross-tenant second-tenant fixture, admin reviewer-transition coverage) — larger, separate efforts.
- Renaming the legacy `Submission.journalMarkdown` UI label ("Engineering journal") to remove the name
  collision with the new dedicated module — a UI copy change, not a docs/testing task; documented as a
  disambiguation note instead.

## Test Scenarios

- **Scenario**: Applicant creates and edits a journal entry against their assigned mission.
  **Actor**: accepted applicant with one assigned mission.
  **Preconditions**: `createSubmissionFixture` (accepted application → assigned Week 1 mission).
  **Steps**: `createJournalEntry`, then `updateJournalEntry` on the same entry.
  **Expected result**: entry appears in `listApplicantJournalEntries`; `journal.created` and
  `journal.updated` audit rows exist; the update persists.
  **Automation**: Added this iteration — `journal` area, `scripts/regression/run.ts`.

- **Scenario**: Applicant cannot create a journal entry against a published mission not assigned to
  them.
  **Actor**: accepted applicant.
  **Preconditions**: a second published mission exists in the same program/week with no
  `MissionAssignment` for this applicant.
  **Steps**: `createJournalEntry` against the unassigned mission.
  **Expected result**: rejected with "Mission is not assigned to this applicant."
  **Automation**: Added this iteration — `journal` area.

- **Scenario**: One journal entry per applicant per calendar date is enforced.
  **Actor**: accepted applicant.
  **Preconditions**: an existing entry for a given date.
  **Steps**: `createJournalEntry` again for the same calendar date (different time-of-day).
  **Expected result**: rejected with `JournalEntryDateConflictError`.
  **Automation**: Added this iteration — `journal` area.

- **Scenario**: Journal entries lock once the mission's assignment is submitted.
  **Actor**: accepted applicant.
  **Preconditions**: an existing journal entry against the assigned mission.
  **Steps**: `saveSubmissionDraft` + `submitSubmission` for that mission, then attempt
  `updateJournalEntry`.
  **Expected result**: `isJournalMissionLockedForApplicant` returns true; the update is rejected with a
  message containing "locked".
  **Automation**: Added this iteration — `journal` area.

- **Scenario**: Applicant mission list/detail and submission drafting are limited to assigned missions.
  **Actor**: accepted applicant with one assigned mission.
  **Preconditions**: a second published, unassigned mission in the same program/week.
  **Steps**: read `listAssignedProgramMissions`/`getAssignedProgramMission` for both missions; attempt
  `saveSubmissionDraft` against the unassigned one.
  **Expected result**: only the assigned mission is visible/accessible; the draft attempt is rejected.
  **Automation**: Added this iteration — `missions` area.

- **Scenario**: An applicant accepted before Mission Assignment existed has no assigned missions.
  **Actor**: applicant with an `Application` inserted directly as `ACCEPTED` (bypassing
  `applyStatusTransition`, simulating pre-existing data).
  **Preconditions**: a published Week 1 mission exists in the program.
  **Steps**: read `listAssignedProgramMissions` for that applicant.
  **Expected result**: empty list — **this documents a known gap**, not desired end-state behavior; the
  scenario will fail loudly (with a message pointing back to this doc) if a future backfill changes
  this, so the change doesn't slip in unnoticed.
  **Automation**: Added this iteration — `missions` area. Deferred: the actual backfill/fix is a
  product decision, tracked in `docs/Regression_Scenarios.md` Known Gaps, not resolved by this patch.

order (before `Mission`/`User`) via the existing `RegressionDataMarker` mechanism, consistent with the
"delete only marker-tagged records" data-ownership rule.
