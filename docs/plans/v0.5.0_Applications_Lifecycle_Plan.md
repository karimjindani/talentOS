# v0.5.0 Applications Lifecycle & Admin Review Plan

Code version: `v0.5.0`

Previous baseline: `v0.4.0`

Previous baseline commit: `48582be01b67839113537ae4bc05c01209b4778a`

Plan status: Implemented

## Summary

Delivers the **Applications-first MVP vertical slice** (Decision Log `D-017`): a real,
tenant-scoped, audited application lifecycle from authenticated apply → submit → admin
review → accept / reject / waitlist. Before `v0.5.0` every read/write of application data was a
stub (inert apply form, stub `POST /api/applications`, hardcoded admin tables, hardcoded
applicant "Draft"). This slice wires the existing Prisma schema, status state-machine and
capability matrix into working applicant and admin flows.

Scope is the **Applications module only**, per the documented cadence: org/user/role management
remains `v0.3.1` (`D-035`) and Programs CRUD is the next MVP module. No schema change or new
migration was required — `Application`, `ApplicationAnswer` and `AuditLog` already existed.

## Decisions

- `v0.5.0` is a minor version (new product capability) per the `Version_Baseline.md` versioning
  convention.
- Apply is **authenticated** via Keycloak (`D-031`/`D-033`/`D-034`); the applicant's DB `User` is
  provisioned/linked by email (backfilling `keycloakSubjectId`) with an `APPLICANT`
  `TenantMembership` on first authenticated action.
- Reviewing enforces the finer `reviewApplications` capability: `ORG_ADMIN`, `HR` and
  `SUPER_ADMIN` may decide; `TECH_LEAD` may enter the portal but cannot decide.
- All writes are tenant-scoped (`assertTenantScopedAccess`) and audited (`AuditLog`).

## Implementation

- **Shared persistence (`packages/db/src/`)** — extracted the Prisma singleton to `client.ts`;
  added pure data-access helpers exported from `index.ts`:
  - `tenants.ts` — `getTenantBySlug`.
  - `programs.ts` — `listPublishedPrograms`.
  - `users.ts` — `provisionApplicantUser`, `getUserByEmail`.
  - `applications.ts` — `findActiveApplication`, `createSubmittedApplication` (txn: application +
    answers + `application.submitted` audit), `listTenantApplications`, `getTenantApplication`,
    `listApplicantApplications`, `applyStatusTransition` (txn: status/`reviewedAt`/notes +
    `application.status_changed` audit).
  Transition and RBAC guards stay in the app layer so the DB package keeps no `@talentos/auth`
  dependency.
- **Workflow helper (`packages/auth/src/workflow.ts`)** — added `nextStatusesFor(status)` (derived
  from `ALLOWED_TRANSITIONS`) to drive the reviewer's valid action buttons.
- **Shared UI (`packages/ui/src/StatusBadge.tsx`)** — presentational status pill reused by both
  portals.
- **Applicant (`apps/applicant`)** — `/apply` now requires auth (added to `PROTECTED_PREFIXES`);
  the apply page is a server component with a `submitApplication` server action (provision user →
  validate published program for the tenant → block duplicate active application → create
  `SUBMITTED` application + motivation answer → redirect to `/application`). `/application` shows
  the applicant's real status, program, timestamps, reviewer notes and answers. The stub
  `app/api/applications/route.ts` was removed.
- **Admin (`apps/admin`)** — `/applications` lists real tenant applications (applicant, program,
  status badge, submitted date) linking to detail; `/applications/[id]` shows answers and a review
  form whose buttons are the valid `nextStatusesFor(current)` transitions. The `reviewApplication`
  server action checks `can("reviewApplications", …)`, resolves the tenant, re-fetches the
  application tenant-scoped, asserts the transition, persists via `applyStatusTransition` (attributing
  the reviewer), and revalidates the list + detail.
- **Seed (`packages/db/prisma/seed.ts`)** — adds one `SUBMITTED` application (with a motivation
  answer) for `applicant@demo.talentos.local`, idempotently, so review is demoable out of the box.

## Tests

Regression suite expanded from 19 to 24 tests: new `packages/auth/src/workflow.test.ts` covers
`nextStatusesFor` (decision options per status, terminal statuses, consistency with
`canTransitionApplicationStatus`, immutability). The existing transition/RBAC/tenant/TOTP tests
remain green. End-to-end apply→review behaviour is verified manually against the Docker stack.

## Out of scope → later
