Latest released baseline at allocation time: `v0.19.3`

## Summary

This iteration restores week-based learning tasks as a track that is deliberately separate from
mission workflow tasks, adds ordered Markdown and YouTube learning resources, and makes completion of
required weekly tasks part of final mission-submission readiness. It also strengthens the dedicated
Engineering Journal experience, requires at least four eligible entries for the current assignment
attempt, validates GitHub, deployment and Loom evidence for public reachability, and preserves
submission and journal state when any readiness or network check fails.

The iteration extends the existing Applicant, Admin and Ops surfaces. It does not replace the Mission
Engine, change assignment selection, or add a new regression framework.

## Objective

Give applicants a clear Week 1 learning path and a single readiness checklist before submission while
ensuring tenant isolation, assignment-attempt journal separation, URL safety and atomic state changes.

## Approved Scope

- Week-scoped `ProgramTask` content, completion and progress.
- Ordered `MARKDOWN` and `YOUTUBE` resources attached to weekly tasks.
- Three seeded required/published Week 1 tasks and Markdown source guides.
- Applicant task/resource pages and idempotent completion controls.
- Engineering Journal field guidance, confidence semantics and future-date rejection.
- Multiple journal entries per week while retaining one entry per applicant calendar date.
- A minimum of four eligible journal entries linked to the current `MissionAssignment` attempt.
- Centralized readiness evaluation for weekly tasks, journals and evidence formats.
- One or more semicolon-separated deployment URLs, normalized into the existing string column.
- Public HTTP(S) reachability checks for GitHub, deployments and Loom, with SSRF defenses.
- Short, status-guarded submission transaction and current-attempt journal locking.
- Repeat/revision compatibility, Admin weekly-content management and regression-dashboard coverage.

## User And Business Requirements

1. Weekly learning tasks belong to a tenant program and week, not to an individual mission.
2. Every published required task for the assigned week must be complete before final submission.
3. A task can provide ordered Markdown guidance and YouTube learning content.
4. Demo seed data supplies three required Week 1 tasks and their source material.
5. The Engineering Journal form explains each field and uses accessible confidence choices.
6. Applicants may create multiple weekly entries on distinct dates, including back-dated entries, but
   may not create future-dated entries or two entries for the same calendar date.
7. Final submission requires at least four eligible entries for the exact current assignment attempt.
8. Evidence includes a GitHub repository, one or more deployed application URLs and a Loom walkthrough.
9. Every evidence URL must be well formed, publicly reachable and safe to request from the server.
10. A failed readiness, reachability or concurrency check must not submit the assignment or lock journals.

## Technical Approach

### Existing Models Reused

- `ProgramTask` remains the tenant/program/week learning-task model and gains `required` and
  `published` flags.
- `VideoResource` keeps its legacy model/table name for compatibility and gains task association,
  `LearningResourceType`, Markdown content, ordering, optional duration and a nullable YouTube URL.
- `UserTaskCompletion` gains authoritative `tenantId`; its unique key is
  `[tenantId, userId, taskId]`, making completion idempotent and reusable across same-week attempts.
- `EngineeringJournalEntry`, `MissionAssignment` and `Submission` are reused. No second journal,
  assignment or deployment-link model is introduced.
- `MissionTaskCompletion` remains the separate per-assignment workflow checklist for Review Brief and
  Study Tutorial. Weekly `ProgramTask` completion does not replace it.

### Readiness And Submission Boundary

`packages/db/src/submission-readiness.ts` derives tenant, applicant, program, week and mission from the
selected assignment. It loads required published weekly tasks, tenant/applicant-scoped completions,
non-future journal rows linked to the exact assignment attempt, and normalized evidence values. Four
or more eligible journals satisfy readiness; older-attempt and future-dated rows do not count.

`packages/db/src/url-safety.ts` parses and normalizes evidence. Deployment input is semicolon-delimited,
trimmed, de-duplicated in input order and capped at ten values. The final public checks occur before the
database transaction. DNS results are screened and pinned, redirects are revalidated, private and
reserved destinations are rejected, and requests have bounded redirects, time and response handling.

After external checks pass, `submitSubmission` opens a short transaction, recomputes critical
readiness, verifies the evidence has not changed, uses status-scoped writes to guard concurrent
submissions, advances the assignment, locks journals for only the exact current attempt, and writes
existing audit/notification records. A failed check exits before those writes.

### Repeat, Revision And Compatibility

- `NEEDS_REVISION` reuses the same submission and attempt and reruns readiness and URL checks.
- `REPEAT` creates a new same-week assignment attempt. Week task completion remains available because
  it is week-level, while the new attempt needs four newly linked journals.
- The existing `Submission.deploymentUrl` string stores normalized values such as
  `https://app.example.com;https://api.example.com`; historical single-URL rows remain valid.
- `Submission.journalMarkdown` remains in the schema for backward compatibility and is not part of the
  dedicated Engineering Journal readiness rule.
- Pending YouTube content is represented by a `YOUTUBE` resource with a null URL, not a fabricated link.

- Program weeks remain numbered consistently with assigned mission weeks.
- The applicant and assignment tenant/program relationships are authoritative.
- One journal row represents one applicant-selected calendar date, normalized by existing journal logic.
- External evidence hosts may be unavailable or rate-limited; failure is treated as a blocker, not success.
- A real TalentOS introduction video URL can be supplied later without a schema change.

## Acceptance Criteria

- Applicants see published tasks/resources for their accepted program week and can complete each task
  idempotently only within their tenant, program and week.
- Final submission is blocked until weekly task, journal and evidence requirements all pass.
- Four means a minimum of four eligible entries, not exactly four.
- Repeat attempts reuse week task completion but cannot count prior-attempt journals.
- Every deployment value displays as a separate safe external link in Applicant and Admin views.
- Failed URL/readiness/concurrency validation causes no status transition and no journal lock.
- Admin content actions retain existing permission and tenant guards.
- Regression scenarios appear under existing dashboard categories.

## Test Scenarios

### Scenario S1: Weekly tasks and ordered resources are scoped to program week

- **Actor**: Accepted applicant
- **Preconditions**: Applicant has an accepted program and active Week 1 assignment.
- **Steps**: Open Tasks and Resources; inspect ordered task resources.
- **Expected result**: Only published tenant/program/week content appears; Markdown is inert and a
  pending YouTube URL is labeled rather than linked.
- **Automation**: Added this iteration (`dashboard.test.ts`, `program-content.test.ts`,
  `SafeMarkdown.test.ts`, Applicant/Admin regression scenarios).

### Scenario S2: Weekly task completion is idempotent and tenant scoped

- **Actor**: Accepted applicant
- **Preconditions**: A required task belongs to the applicant's active program week.
- **Steps**: Complete the same task twice and attempt a cross-tenant/program/week task.
- **Expected result**: One completion exists; invalid scope is rejected before a write.
- **Automation**: Added this iteration (`dashboard.test.ts`, Applicant and Tenant regressions).

### Scenario S3: Journal dates and structured fields are validated

- **Actor**: Accepted applicant
- **Preconditions**: Applicant has an editable assigned mission.
- **Steps**: Create entries on distinct current/back dates; try a future date and duplicate date.
- **Expected result**: Valid dates and fields save; future/duplicate dates fail clearly.
- **Automation**: Added/updated this iteration (`journal.test.ts`, Applicant regression).

### Scenario S4: Readiness counts four journals from only the current attempt

- **Actor**: Accepted applicant
- **Preconditions**: Current and previous same-week assignment attempts exist.
- **Steps**: Add journals to the old attempt and fewer than four to the current attempt; check readiness.
- **Expected result**: Old/future rows do not count; readiness passes at four current eligible rows.
- **Automation**: Added this iteration (`submission-readiness.test.ts`, Missions and Tenant regressions).

### Scenario S5: Evidence parsing supports one or more deployment URLs

- **Actor**: Accepted applicant
- **Preconditions**: Editable submission draft.
- **Steps**: Enter GitHub and Loom links plus semicolon-separated deployment links with spaces,
  duplicates and blanks.
- **Expected result**: Values normalize, exact duplicates are removed in order, and single-URL history
  remains compatible.
- **Automation**: Added this iteration (`url-safety.test.ts`, `submission-readiness.test.ts`,
  `submissions.test.ts`).

### Scenario S6: Unsafe or unreachable evidence is rejected per URL

- **Actor**: Accepted applicant
- **Preconditions**: Readiness otherwise passes.
- **Steps**: Supply malformed, private, redirect-to-private, timed-out or unreachable evidence.
- **Expected result**: The exact failing URL is reported and submission is blocked.
- **Automation**: Added this iteration (`url-safety.test.ts`, `submission-readiness.test.ts`, Missions regression).

### Scenario S7: Failed submission checks do not change durable state

- **Actor**: Accepted applicant
- **Preconditions**: Draft submission and unlocked current-attempt journals.
- **Steps**: Submit with incomplete readiness or one failed public URL.
- **Expected result**: Submission and assignment status stay unchanged; no journal receives `lockedAt`.
- **Automation**: Added this iteration (`submissions.test.ts`, Missions regression).

### Scenario S8: Successful submission locks only current-attempt journals

- **Actor**: Accepted applicant
- **Preconditions**: All weekly, journal, mission-task and evidence checks pass.
- **Steps**: Submit once and retry the same submission concurrently/repeatedly.
- **Expected result**: One status transition occurs and only exact-attempt journals lock.
- **Automation**: Added/updated this iteration (`submissions.test.ts`, Missions regression).

### Scenario S9: Revision and repeat attempts remain separated

- **Actor**: Applicant and reviewer
- **Preconditions**: A submitted assignment can be revised or repeated.
- **Steps**: Request revision, resubmit, then exercise repeat behavior on a separate fixture.
- **Expected result**: Revision reuses the attempt; repeat creates one same-week attempt, preserves old

## Security And Data Rules

- Every task, completion, journal, assignment, submission and admin-content query is tenant scoped.
- Applicant writes additionally prove accepted-program membership and active assignment ownership.
- User input never supplies authoritative week/program/mission relationships for readiness.
- Markdown rendering does not enable raw HTML execution.
- External requests allow only public HTTP(S), reject credentials/private/reserved destinations,
  revalidate redirects and pin validated DNS addresses.
- Network I/O is kept outside the database transaction; critical state is rechecked inside it.
- Status-scoped writes and evidence rechecks protect against concurrent/double submissions.
- See `docs/audits/v0.19.5_Weekly_Tasks_Submission_Readiness_Security_Review.md`.
