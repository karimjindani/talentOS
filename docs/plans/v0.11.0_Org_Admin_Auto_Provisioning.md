# v0.11.0 — Org-Admin Auto-Provisioning via Keycloak Admin REST API (D-053)

Code version: `v0.11.0` · Previous: `v0.10.4` · Date: `2026-07-02`

## Problem

Creating an organization only wrote the DB (`Tenant` + `User` + `ORG_ADMIN` `TenantMembership` + audit);
the org admin had **no Keycloak identity and no realm role**, so they could not sign in until someone ran
`kcadm` by hand. This delivers the deferred Keycloak-Admin-API provisioning (D-035 / backlog "v0.3.1").

## Fix (code-only, no DB schema change)

- **Service-account client** — `keycloak/import/talentos-realm.json` adds confidential
  `talentos-provisioner` (`serviceAccountsEnabled`, realm-management `manage-users`/`view-realm`/
  `query-users`). Fresh envs get it from the import; a running realm is patched live via `kcadm.sh`.
- **Provisioning module** (server-only) — `apps/admin/lib/keycloak-admin.ts`: `getAdminToken()`
  (client_credentials), `provisionOrgAdmin({ email, name })` (find-or-create user with `emailVerified` +
  required actions `UPDATE_PASSWORD`/`CONFIGURE_TOTP`, set a generated one-time temp password for new
  users, always ensure the `ORG_ADMIN` realm role; idempotent), plus pure helpers `generateTempPassword`
  and `adminBaseFromIssuer`. Modeled on the token-POST in `apps/ops/src/auth.ts`. Never imported by edge
  middleware.
- **Wire-in** — `apps/admin/app/organizations/actions.ts`: after `createOrganization`, call
  `provisionOrgAdmin`; the action now returns a typed `OrgActionState` (`useActionState`) instead of
  throwing. Keycloak failure does not roll back the DB org; the message is retryable (provisioning is
  idempotent).
- **UI** — new `apps/admin/app/organizations/CreateOrgForm.tsx` (`"use client"`, first `useActionState` in
  the admin app) renders success + the **one-time temp password**, or an inline error; the old amber
  "assign the realm role manually" notice is gone.
- **Config** — `KEYCLOAK_SERVER_URL`, `KEYCLOAK_REALM`, `KEYCLOAK_PROVISIONER_CLIENT_ID/SECRET` added to
  `.env`, `.env.example`, and the admin service in `docker-compose.yml`.

## Decisions
Service-account client (least-privilege, no master creds in app); one-time temp password shown once (no
SMTP), forcing reset + TOTP on first login. See `D-053`.

## Verification
`db:generate` + `typecheck` + `lint` + `test` green (**71 tests**, new
`apps/admin/lib/keycloak-admin.test.ts`). Live E2E against the running Keycloak (service-account patched
in): a new email was provisioned (user created, `emailVerified=true`, required actions set, `ORG_ADMIN`
granted, temp password returned) and an existing user re-provisioned idempotently (no password, role
ensured). Admin rebuilt (exit 0); portal `307`. Full detail:
`docs/testing/v0.11.0_Org_Admin_Auto_Provisioning_Test_Results.md`.

## Out of scope
Full user-management UI (list/edit/deactivate); SMTP/email reset; resetting an existing user's password;
reserved-slug blocklist / `Application` unique constraint (#4).
