# v0.17.1 Journal Entry Date Uniqueness Fix

Code version: `v0.17.1`

Previous baseline: `v0.17.0`

Plan status: Implemented (retroactively documented — see note below)

## Summary

Patch on top of `v0.17.0` (Engineering Journal MVP). The `v0.17.0` plan already stated the rule "each
applicant can have only one entry per calendar date within a tenant," but the initial migration
(`20260707190000_v0_17_0_engineering_journal_mvp`) only created a non-unique index on
`(tenantId, applicantId, entryDate)` — the rule was enforced in application code
(`packages/db/src/journal.ts`) but not at the database layer. This patch closes that gap with a real
unique constraint.

## Decisions

- Keep the `v0.17.0` baseline; record this as patch `v0.17.1` (Version_Baseline convention: a small,
  non-breaking data-integrity fix on an already-shipped model, not a new capability).
- Enforce uniqueness in Postgres, not only in application code, so the rule holds even if a future
  caller bypasses `packages/db/src/journal.ts`.
- Normalize `entryDate` to a calendar day (`date_trunc('day', ...)`) before adding the constraint, so
  any existing rows carrying a time component collapse onto the same key as new writes (which already
  save entry dates without a time component).

## Implementation

- Migration `20260708100000_v0_17_1_journal_entry_date_unique`
  (`packages/db/prisma/migrations/20260708100000_v0_17_1_journal_entry_date_unique/migration.sql`):
  1. `UPDATE engineering_journal_entries SET "entryDate" = date_trunc('day', "entryDate")` — normalizes
     any pre-existing rows.
  2. `CREATE UNIQUE INDEX engineering_journal_entries_tenantId_applicantId_entryDate_key ON
     engineering_journal_entries("tenantId", "applicantId", "entryDate")`.
  3. Drops the superseded non-unique index
     (`engineering_journal_entries_tenantId_applicantId_entryDate_idx`) added in `v0.17.0`.
- `packages/db/prisma/schema.prisma` — `EngineeringJournalEntry` gains
  `@@unique([tenantId, applicantId, entryDate])` in place of the plain index.
- No application code change: `packages/db/src/journal.ts` already checked for an existing same-day
  entry before insert; the constraint is a defense-in-depth backstop, and its violation now surfaces as
  a Prisma `P2002` the same way other unique-constraint checks in this codebase are handled.

## Documentation note

This migration and fix shipped as part of the same implementation commit as `v0.17.0` and `v0.18.0`
(`c7413eb`, branch `engineering-journal-mvp`) without a dedicated plan/test-results doc at the time,
which is a deviation from the one-plan-doc-per-version rule in `docs/sdlc.md`. This doc and
`docs/testing/v0.17.1_Journal_Entry_Date_Unique_Test_Results.md` were added retroactively during the
pre-push SSDLC compliance pass to close that gap; see `D-074` in `docs/Decision_Log.md`.

## Tests

No new dedicated unit test file; covered by the existing `packages/db/src/journal.test.ts` same-day
uniqueness cases (which already asserted the applicant-facing rule via application-level checks) and by
the full regression suite. See verification below.
