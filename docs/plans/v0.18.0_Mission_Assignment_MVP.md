# v0.18.0 Mission Assignment MVP

## Goal

Give each accepted applicant an assigned Week 1 mission instead of showing every published mission in
the accepted program. Seed multiple Week 1 mission variants from Markdown source files so future AI
review workflows can preserve assignment context.

## Scope

- Add a `MissionAssignment` table linking tenant, program, applicant, mission and week.
- Enforce one mission assignment per tenant + program + applicant + week.
- Assign one Week 1 published mission when an application becomes `ACCEPTED`.
- Keep assignment creation idempotent so repeated acceptance handling does not duplicate rows.
- Seed multiple Week 1 mission variants from Markdown files under
  `packages/db/prisma/seed-data/missions/ai-native-engineering/week-1/`.
- Import Markdown specs into existing `Mission` fields during seed; do not depend on Markdown paths at
  runtime.
- Limit applicant mission list, mission detail access, submission drafts and journal mission choices to
  assigned missions.

## Out of Scope

- Auth, Keycloak, permission matrix or admin permission changes.
- Mission submission workflow state changes.
- Recruiter or portfolio pages.
- Real AI assignment review or scoring.
- Runtime loading of Markdown assignment files.
- Engineering Journal behavior beyond using assigned missions for mission selection.

## Safety Notes

The change is additive at the database level. Existing `Mission` and `Submission` records remain in
place. The main behavior change is applicant visibility: a published mission is now eligible for
assignment, while an applicant must have a `MissionAssignment` row to see, journal against or submit for
that mission.
