# v0.19.6 Mission Workspace LMS, Curriculum Tooling And Thursday Scheduling

Code version: `v0.19.6`

Previous baseline: `v0.19.5`

Plan status: Implemented

## Summary

This iteration modernizes the applicant **Mission Workspace** into a tabbed, LMS-style experience and
extends the admin curriculum tooling and mission scheduling around it, while preserving every existing
mission/assignment/submission business rule. The workspace becomes one page with a compact header, a
step/task tab rail, in-tab YouTube playback gated to 90% watched, sequentially-unlocked weekly learning
tasks, and a live `d h m s` deadline countdown. Admins gain a top-level **Tasks** page (searchable
program picker, collapsible task cards, inline resource management), **document** learning resources
(real file upload), **prerequisite** tasks that lock the mission's own steps until complete, list
**pagination + filters** across Applications/Programs/Missions/Submissions, and a live **Overview KPI
dashboard**. Mission scheduling changes so every applicant gets a Thursday deadline with at least four
working days, and a repeat never re-serves a mission the applicant already had.

## Scope

- Applicant Mission Workspace rebuilt as a tabbed LMS layout (Overview + step tabs + learning-task tabs
  + Submission), compact navy header, live countdown, and Continue navigation, reusing all existing
  `@talentos/db` reads/actions and gating.
- Weekly learning tasks surfaced as workspace tabs with inline resources; YouTube plays in-tab and
  "Mark complete" unlocks only after ≥90% watched; learning tasks unlock sequentially.
- `DOCUMENT` learning-resource type: admins upload a file (PDF/DOC/DOCX/TXT/image) via the existing
  presign→storage→confirm flow; applicants download it through a gated learning-resource route.
- `isPrerequisite` program-task flag: incomplete prerequisites lock the mission's own steps (Review
  Brief / Study Tutorial / Submission) after acceptance.
- Admin top-level **Tasks** page: searchable program selector, `Task N` collapsible cards, inline
  add/edit resources (Markdown/YouTube/Document) with auto-collapse after save.
- Admin **Overview** KPI dashboard from live tenant counts; list **pagination (10/20 rows) + filters**
  for Applications, Programs, Missions, Submissions.
- Shared UI primitives in `@talentos/ui` (`Card`, `SectionCard`, `ProseText`, `ProgressBar`, `Button`,
  `MissionStatusBadge`, `SubmissionStatusBadge`) replacing duplicated inline markup and status maps.
- Mission scheduling: deadline follows a **Thursday cadence with ≥4 working days** instead of raw
  `deadlineHours`; a **repeat excludes every mission the applicant already had**, not only the failed one.

## Out of Scope

- Regenerating the mission engine, assignment selection, RBAC, Keycloak, or submission state machine.
- Server-side pagination (list pages paginate in memory; adequate at current admin volumes).
- Dark mode for the applicant portal (light-only retained).
- Per-tenant timezone for the Thursday cadence (computed in UTC; a tenant timezone is future work).
- Automated browser (DOM) assertions for the new client components (video gate, tab switching,
  collapsibles) — covered by extracted view-model/pure-logic unit tests; DOM-level checks deferred.

## Test Scenarios

### Scenario S1: Mission Workspace derives steps, progress and submission mode

- **Actor**: Accepted applicant
- **Preconditions**: An accepted mission assignment with the fixed 3 mission steps.
- **Steps**: Open the mission; observe tab rail, header progress, and which submission UI shows.
- **Expected result**: Steps map to complete/current/upcoming; progress = completed/total; the
  submission section shows accept / failed / editable / locked exactly as the prior page did.
- **Automation**: Added this iteration (`apps/applicant/app/dashboard/missions/[id]/view-model.test.ts`).

### Scenario S2: Accepting a mission sets a Thursday deadline with ≥4 working days

- **Actor**: Accepted applicant
- **Preconditions**: A `NOT_STARTED` assignment.
- **Steps**: Accept the mission on any weekday.
- **Expected result**: `deadlineAt` falls on a Thursday (UTC) with at least four Mon–Thu working days
  from acceptance; `graceEndsAt` = deadline + grace hours.
- **Automation**: Added this iteration (`packages/db/src/mission-assignments.test.ts` `computeMissionDeadline`
  7-day cases + accept test; Missions regression "Accepting a mission sets a Thursday deadline…").

### Scenario S3: A repeat never re-serves a previously-assigned mission

- **Actor**: Reviewer + applicant
- **Preconditions**: A submitted attempt and ≥1 alternate published Week-N mission.
- **Steps**: Review `REPEAT`; on a second repeat, inspect which mission is assigned.
- **Expected result**: The new attempt excludes every mission the applicant already had this week
  (`id: { notIn: [...all prior] }`); when none remain, the application moves to
  `AWAITING_MISSION_ASSIGNMENT` and reviewers are notified.
- **Automation**: Added/updated this iteration (`mission-assignments.test.ts` repeat test; Missions
  repeat regression scenarios).

### Scenario S4: Prerequisite tasks lock the mission's steps until complete

- **Actor**: Accepted applicant
- **Preconditions**: The mission's week has an `isPrerequisite` published task.
- **Steps**: Accept the mission; open the step tabs while the prerequisite is incomplete, then complete it.
- **Expected result**: Review Brief / Study Tutorial / Submission tabs are locked (🔒) with a
  "complete the prerequisite" prompt until all prerequisites are complete, then unlock.
- **Automation**: Persistence added this iteration (`program-content.test.ts`; Missions regression
  "Prerequisite weekly tasks are stored…"). Applicant-side step-lock UI: Deferred (Known Gap).

### Scenario S5: Document learning resources upload and download safely

- **Actor**: ORG_ADMIN uploads; accepted applicant downloads
- **Preconditions**: A program with a weekly task; `manageProgramContent` for the admin.
- **Steps**: Add a `DOCUMENT` resource (upload a PDF); as an applicant, open the task and download it.
- **Expected result**: The resource stores a validated in-tenant `fileId`; a `DOCUMENT` with no file, or
  a file from another tenant, is rejected; the applicant download route only serves
  `learning-resource`-category files of the applicant's tenant.
- **Automation**: Added this iteration (`program-content.test.ts` DOCUMENT test). Applicant download
  route + upload UI: Deferred (Known Gap).

### Scenario S6: YouTube learning resource gates completion at 90% watched

- **Actor**: Accepted applicant
- **Preconditions**: A learning task tab with a YouTube resource.
- **Steps**: Open the task tab; attempt "Mark complete" before and after watching ≥90%.
- **Expected result**: "Mark complete" is disabled with a reason until every video is ≥90% watched;

## Security and Data Rules

- Every new query/action is tenant-scoped; admin content writes keep the existing `manageProgramContent`
  guard; document upload uses the existing authenticated presign/confirm routes.
- Document resources link only a `StoredFile` verified to belong to the tenant; the applicant download
  route presigns only `category = "learning-resource"` files of the applicant's tenant (never CVs).
- The Thursday deadline and prerequisite gating add no new write authority; prerequisite step-locking is
  a UI gate over unchanged completion/submission rules.
- No change to the submission state machine, URL-safety checks, or journal locking from `v0.19.5`.
