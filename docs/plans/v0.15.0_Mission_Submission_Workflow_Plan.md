# v0.15.0 Mission Submission Workflow Plan

Code version: `v0.15.0`

## Summary

Implement Mission Submission MVP-1 (D-067), the evidence-collection half of the SEM learning loop on
top of the `v0.14.0` Mission Engine. Accepted applicants submit evidence for published missions of
their accepted program — Git repository URL, deployed-application URL, Loom walkthrough URL and an
inline Engineering Journal (Markdown), matching the Week 1 deliverables in `docs/curriculum.md` (PRD,
README, user stories and acceptance criteria live inside the repository). Staff reviewers accept the
submission or request changes with written feedback, and the applicant revises and resubmits — the
SEM coaching loop. Per `docs/Graduate_Profile.md`, graduates are not code reviewers, so there is no
peer review: only ORG_ADMIN and TECH_LEAD (and SUPER_ADMIN) review. An ACCEPTED submission is
terminal and constitutes portfolio/graduation evidence for the mission's `competencyTags`.

## Key Changes

- **Schema** (migration `20260706090000_v0_15_0_mission_submissions`): `Submission` (unused v0.14.0
  scaffolding) gains `tenantId` (direct tenant scoping + FK, backfilled from the parent mission),
  `reviewerFeedback`, `reviewedAt`, `reviewerUserId` (FK → `users`, SetNull), a
  `@@unique([missionId, applicantId])` (one row per applicant per mission; the revision loop reuses
  it) and `@@index([tenantId, status])`. Also drops the leftover init-migration
  `missions_tenantId_programId_idx` superseded in v0.14.0. The unused `REVIEWED` enum value stays.
- **`packages/auth`**: new `reviewSubmissions` capability (ORG_ADMIN + TECH_LEAD; SUPER_ADMIN
  bypass); submission status machine in `workflow.ts` (`DRAFT→SUBMITTED`,
  `SUBMITTED→ACCEPTED|NEEDS_REVISION`, `NEEDS_REVISION→SUBMITTED`) with
  `assertSubmissionStatusTransition` / `nextSubmissionStatuses` / `isSubmissionEditable`.
- **`packages/db/src/submissions.ts`** (new): tenant-scoped helpers mirroring `missions.ts` —
  `getApplicantSubmission`, `listApplicantProgramSubmissions`, `listMissionSubmissions`,
  `getTenantSubmission`, `saveSubmissionDraft` (upsert, PUBLISHED-mission + editable-status guards),
  `submitSubmission` (ownership + ≥1 evidence link), `reviewSubmission` (SUBMITTED-only, sets
  reviewer fields, audits and notifies the applicant in one transaction). Audit actions:
  `submission.created`, `submission.updated`, `submission.submitted`, `submission.reviewed`.
  Evidence URLs are validated by `parseEvidenceUrl` — repository host-allowlisted to `github.com`,
  Loom to `loom.com`, deployment any http(s) — mirroring the apply flow's profile-link rule.
- **Applicant portal**: mission detail page gains a **My Submission** section (status badge,
  reviewer-feedback panel, evidence form with Save draft / Submit via `useActionState`; read-only
  locked view once SUBMITTED/ACCEPTED); `saveSubmissionAction` guarded by
  `requireTenantAccess("accessApplicantPortal")` + accepted-application → published-mission chain.
  The missions list shows per-mission submission status chips.
- **Admin portal**: mission detail gains a **Submissions** table (all admin roles see it); new
  review page `/missions/[id]/submissions/[submissionId]` shows evidence links, journal, competency
  tags and prior review, with Accept / Request-changes actions rendered only for
  `can("reviewSubmissions")` (HR read-only). `reviewSubmissionAction` is guarded by
  `requireTenantAccess("reviewSubmissions")` + `assertTenantScopedAccess` +
  `assertSubmissionStatusTransition`; written feedback is mandatory for NEEDS_REVISION.
- **Notifications**: applicant receives SUCCESS on acceptance, WARNING with feedback on revision
  request (existing `/dashboard/notifications` surfaces them).
- **Regression**: `Submission` added to `RegressionEntityType` + FK-safe cleanup order; new
  scenarios — full submission loop with notifications/audit/terminal-acceptance, the
  `reviewSubmissions` role matrix, and a cross-tenant submission-read isolation check.

## Test Plan

- Unit tests: submission transition machine + editability (`packages/auth/src/workflow.test.ts`),
  `reviewSubmissions` role matrix (`permissions.test.ts`), evidence-URL allowlists and all helper
  guards (`packages/db/src/submissions.test.ts`).
- Scenario regression: `regression:missions` (submission loop, role matrix) and `regression:tenant`
  (cross-tenant read).
- Browser e2e: accepted applicant drafts → submits on the Week 1 mission; TECH_LEAD requests
  changes with feedback; applicant sees WARNING notification + feedback, revises, resubmits;
  reviewer accepts; SUCCESS notification; HR sees read-only review page.
- Required validation: `npm.cmd run db:generate`, `npm.cmd run db:migrate`, `npm.cmd run typecheck`,
  `npm.cmd run test`, `npm.cmd run build`, `npm.cmd run lint`, `docker compose config --quiet`,
  `npm.cmd run regression:missions`, `npm.cmd run regression:all`,
  `npm.cmd run ops:cleanup-regression`.

## Assumptions

- Version is `v0.15.0`; decision `D-067`.
- Evidence is URLs + journal markdown only; file attachments (MinIO) are deferred.
- One submission row per applicant per mission; ACCEPTED is terminal.
- Engineering Journal remains an inline field; a dedicated journal module is a later slice.
- Competency rollup / portfolio view is deferred — this slice keeps
  `accepted submission → mission.competencyTags` queryable for it.
