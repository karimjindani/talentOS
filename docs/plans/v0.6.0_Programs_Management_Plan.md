# v0.6.0 Programs Management Plan

Code version: `v0.6.0`

Previous baseline: `v0.5.0`

Previous baseline commit: `f09e30b`

Plan status: Implemented

## Summary

Delivers the **Programs** MVP module (`Product_Backlog.md`; `Architecture.md` Engineering To-Do item 4):
tenant-scoped, capability-gated, audited admin CRUD for programs (create / edit / publish / archive).
Before `v0.6.0`, programs were seed-only and `apps/admin/app/programs/page.tsx` was a static stub. The
applicant apply form already consumes published programs (`listPublishedPrograms`), so this slice closes
the loop: admins can now create and publish the programs applicants apply to. No schema change was
required — the `Program` model already existed.

## Decisions

- `v0.6.0` is a minor version (new product capability) per the `Version_Baseline.md` convention.
- Mutations require the `managePrograms` capability (`ORG_ADMIN` / `SUPER_ADMIN`); `HR` / `TECH_LEAD`
  may view the list but cannot create/edit/transition (capability matrix in
  `packages/auth/src/permissions.ts`).
- Program status uses a light state machine (`DRAFT ⇄ PUBLISHED ⇄ ARCHIVED`); the UI only offers valid
  transitions, mirroring the application lifecycle.
- Slug is auto-derived from the name with manual override; unique per tenant (duplicate slug surfaces a
  friendly error).
- Optional cohort dates (`startsAt` / `endsAt`) are editable.
- All writes are tenant-scoped (`assertTenantScopedAccess`) and audited (`program.created`,
  `program.updated`, `program.status_changed`).

## Implementation

- **Program state machine (`packages/auth/src/workflow.ts`)** — `ProgramStatus`,
  `canTransitionProgramStatus`, `assertProgramStatusTransition`, `nextProgramStatuses`, unit-tested.
- **Shared UI (`packages/ui/src/StatusBadge.tsx`)** — added `PUBLISHED` / `ARCHIVED` styles.
- **Persistence (`packages/db/src/programs.ts`)** — `listTenantPrograms`, `getTenantProgram`,
  `createProgram` (+`program.created` audit), `updateProgram` (+`program.updated` audit),
  `setProgramStatus` (+`program.status_changed` audit), and `slugify`; duplicate `[tenantId, slug]`
  mapped to a clear error. `listPublishedPrograms` retained as the single source for the apply form.
- **Admin UI (`apps/admin/app/programs/`)** — real list with status badges and (manager-only)
  New/Edit affordances; `new/page.tsx` create form; `[id]/page.tsx` edit form + status-transition
  buttons. Shared server actions in `actions.ts` (`createProgramAction`, `updateProgramAction`,
  `setProgramStatusAction`) each enforce `managePrograms`, resolve the tenant, scope the write,
  attribute the actor, assert the transition and `revalidatePath`.
- **Seed (`packages/db/prisma/seed.ts`)** — added an idempotent `DRAFT` program alongside the published
  one.

## Tests

Regression suite expanded from 24 to 28 tests: `workflow.test.ts` now covers the program state machine
(valid/invalid transitions, `nextProgramStatuses`). End-to-end persistence/audit verified by a lifecycle
integration script against the running database (create → update → publish → archive, each audit row),
plus manual admin/applicant UI verification.
