# v0.19.1 Dashboard Wiring & Same-Week Repeat

Code version: `v0.19.1`

Previous baseline: `v0.19.0`

Plan status: Implemented

## Summary

This patch closes two gaps left by `v0.18.5`/`v0.19.0`: several applicant dashboard surfaces
(Dashboard, My Program, Tasks, Missions) still showed program-level or placeholder figures instead
of the real mission/deadline/task data those versions introduced, and the `REPEAT` review decision
reassigned the applicant back to Week 1 regardless of which week actually failed. Both are wiring/
behavior corrections over already-shipped models — no new schema.

## Scope

- **Dashboard** "Days Remaining" now derives from the applicant's current mission assignment's
  `deadlineAt`, not the program's `endsAt`; the Current Mission card shows the live
  `DeadlineCountdown`.
- **My Program** page now derives Overall Progress, per-week completion and dates from
  `getApplicantMissionProgress` + `listAssignedMissionsWithTasks`: Start Date = the Week 1
  assignment's `acceptedAt`; End Date = Start Date + `PROGRAM_WEEKS` (4) weeks; the current week
  shows the live countdown.
- **Missions** list and **Tasks** page show the live `DeadlineCountdown` for the current
  (accepted/in-progress) mission only — never on a mission the applicant hasn't accepted yet or has
  already submitted/passed/failed.
- `DeadlineCountdown` placement is confirmed correct as shipped in `v0.19.0` — it belongs on the
  mission detail page and the list/tasks/program surfaces, not duplicated inside the mission brief
  content itself.
- **Reject-reassignment correction**: `REPEAT` now reassigns the applicant a different published
  mission for the **same week that failed**, never resetting to Week 1 regardless of which week the
  applicant was on. `createRepeatFromWeekOneTx` is renamed `createRepeatMissionForSameWeekTx` and
  takes the failed assignment's `weekNumber` instead of assuming `1`. The no-alternate-mission
  fallback (`AWAITING_MISSION_ASSIGNMENT` + reviewer notification) is unchanged in behavior, only
  now scoped to the correct week.

## Out of Scope

- Any change to the deadline/grace/auto-advance/disqualification logic itself (`v0.18.5`) —
  unchanged in this patch.
- Any change to the task template or YouTube watch-gate (`v0.19.0`) — unchanged in this patch.
- A rejoin-from-Week-1 path for disqualified applicants — still explicitly deferred.

## Test Scenarios

- **Scenario**: Dashboard Days Remaining reflects the real current-mission deadline.
  **Actor**: accepted applicant with an accepted, unsubmitted current-week mission.
  **Preconditions**: applicant has accepted a mission with a known `deadlineAt`.
  **Steps**: applicant opens `/dashboard`.
  **Expected result**: the Days Remaining stat matches the time remaining until the current
  mission's `deadlineAt`, not the program's `endsAt`.
  **Automation**: Deferred — verified manually in browser; no automated scenario asserts the
  dashboard stat's source field.

- **Scenario**: My Program shows real start/end dates and per-week completion.
  **Actor**: accepted applicant partway through the 4-week arc.
  **Preconditions**: Week 1 assignment has an `acceptedAt`; at least one week has an accepted
  submission.
  **Steps**: applicant opens `/dashboard/program`.
  **Expected result**: Start Date equals the Week 1 assignment's `acceptedAt`; End Date equals Start
  Date + 4 weeks; completed weeks show 100%, the current week shows its live countdown.
  **Automation**: Deferred — verified manually in browser.

- **Scenario**: Deadline countdown appears only on the current, unsubmitted mission.
  **Actor**: accepted applicant with a mix of `NOT_STARTED`, current, and `PASSED` missions across
  weeks.
  **Preconditions**: at least one past (passed) week and one current (accepted, unsubmitted) week.
  **Steps**: applicant opens Missions, Tasks and My Program.
  **Expected result**: the countdown renders next to the current mission only — never for a
  not-yet-accepted or already-resolved (passed/failed) mission.
  **Automation**: Deferred — verified manually in browser across all three pages.

- **Scenario**: Rejection repeats the same week, not Week 1.
  **Actor**: ORG_ADMIN/TECH_LEAD reviewer rejecting a Week 3 submission.
  **Preconditions**: applicant is on a Week 3 mission attempt; an alternate published Week 3 mission
  exists.
  **Steps**: reviewer issues a `REPEAT` decision.
  **Expected result**: the applicant's new assignment is for Week 3 (a different mission than the
  one just failed) — not Week 1.
  **Automation**: Added this iteration — `packages/db/src/mission-assignments.test.ts` gained a
  dedicated "repeats week 3, not week 1" unit test; the existing `missions`-area regression
  scenarios "Repeat-week attempts preserve journal history without duplicate or infinite loops" and
  "Repeated-week history stays separate across mission variants and attempt boundaries" already
  assert the replacement assignment keeps the failed assignment's `weekNumber`, which now also
  covers the corrected same-week behavior end-to-end.

- **Scenario (negative/edge)**: No alternate mission for the failed week still notifies reviewers,
  scoped to the correct week.
  **Actor**: reviewer rejecting a Week 2 submission with no alternate Week 2 mission published.
  **Preconditions**: only one published mission exists for Week 2.
  **Steps**: reviewer issues a `REPEAT` decision.
  **Expected result**: `AWAITING_MISSION_ASSIGNMENT` + reviewer notification, with the notification
  text naming Week 2 (the actual failed week), not Week 1.
  **Automation**: Deferred — covered at unit level only (`mission-assignments.test.ts`); no
  dedicated regression-runner scenario asserts the notification text's week number.

## Security and Data Rules

No new access-control surface; all writes continue to go through the existing tenant/applicant/
assignment-scoped helpers from `v0.18.0`/`v0.18.5`.
