# v0.11.2 — Engineering Governance Documentation — Implementation Plan

Code version: `v0.11.2`

Baseline commit: `7bc6d5e`

Previous baseline: `v0.11.1`

## Objective

Close the engineering-governance gaps in the SSDLC documentation. `docs/sdlc.md` codified the product
SSDLC *principles* but was silent on **how engineering is operated** — source control and CI/CD. The
repository already followed a consistent workflow and had a CI pipeline, but none of it was written down
(violating principle 0, "Always document what you do"); the CD path genuinely does not exist yet. This
baseline documents and formalizes both areas. **Documentation-only** — no application code, pipeline, or
schema change.

## Scope

Two batches of gaps, **policy-docs only** (no `ci.yml`/pipeline code changes this iteration):

**Batch A — Source control & branching:** branching strategy, PR/code-review policy, commit-message
standard, merge/rebase policy, protected-branch/merge-freeze rules.

**Batch B — CI/CD & delivery:** document the existing CI gate + specify the missing security-scan;
define CD, image versioning/registry, dev→staging→prod promotion, and rollback (all as design targets).

## Design decisions

- **Codify reality, not an ideal.** The branching/commit/PR conventions are transcribed from actual git
  history (`<type>/vX.Y.Z-<slug>` branches, Conventional Commits with `(vX.Y.Z, D-0NN)`, PR-merge into
  `main`, the five-job `ci.yml` gate) so the policy matches what the team does.
- **Policy-first for CD.** Per the approved scope, the security-scan stage, image build/push, and deploy
  job are documented as design targets; `ci.yml` is untouched. This puts the gaps on record (principle
  7) without a risky pipeline change in a docs baseline.
- **Enforcement is out-of-repo.** Branch protection and CI/deploy secrets can't be committed; they are
  documented as checklists to enable in the GitHub UI.

## Out of scope (deferred)

- Implementing the scan stage, image registry/push, and CD deploy workflow.
- Other governance domains sdlc.md is silent on (security ops, observability/incident response, release
  management, backup/DR, environments & secrets, Definition of Done).
- The separate automated-test-coverage gap (untested `tenant-guard`, `db/tenants`/`programs`,
  integration tests).
