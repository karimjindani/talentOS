# v0.2.0 Module Isolation Plan

Code version: `v0.2.0`

Previous baseline: `v0.1.1`

Previous baseline commit: `3cb7dcf78e56a9c916a02216763e0c0f2cd27c32`

Implementation commit: `fc0a690b71ef01cc07a3e5711d163040296f6a96`

Plan status: Implemented

## Summary

Isolate the applicant module and the administrator module into two separate Next.js applications, each
built into its own Docker container, sharing only the `packages/*` libraries.

Previously both modules lived in a single application (`apps/web`) and ran as one `web` container, so
they could not be deployed, scaled, or secured independently and shared a single process/attack surface.

This is a structural and deployment change. There is **no Prisma schema or data-model change** in this
iteration.

## Design Decisions

- Admin URLs move to the **root** of the admin container (`/`, `/applications`, `/applications/[id]`,
  `/programs`, `/settings`) — no `/admin` prefix on a dedicated host.
- Shared front-end code is extracted into a shared package (`packages/ui`) rather than duplicated.

## Target Architecture

| Module | App | Container | Host port | Internal port |
| --- | --- | --- | --- | --- |
| Applicant | `apps/applicant` (renamed from `apps/web`) | `talentos-applicant` | `3100` (`APPLICANT_PORT`) | `3000` |
| Administrator | `apps/admin` (new) | `talentos-admin` | `3200` (`ADMIN_PORT`) | `3000` |
| Database | `packages/db` | `talentos-postgres` | `55432`/`5432` (`POSTGRES_PORT`) | `5432` |

Both apps depend on `packages/auth`, `packages/db`, and the new `packages/ui`. The AI mentor route is
applicant-facing and stays in the applicant app.

## Implementation

- New `packages/ui` (`@talentos/ui`): shared `StatusCard`, `getTenantContext`, and a Tailwind brand
  preset consumed by both apps.
- `apps/web` renamed to `apps/applicant` (`@talentos/applicant`); admin routes removed; shared imports
  point at `@talentos/ui`; the `Admin` link targets `NEXT_PUBLIC_ADMIN_URL`.
- New `apps/admin` (`@talentos/admin`): admin pages moved to the root, sidebar nav in the root layout,
  "Applicant portal" link via `NEXT_PUBLIC_APPLICANT_URL`.
- One parameterized root `Dockerfile` (build args `APP_NAME` / `APP_DIR`) builds either container; the
  old `apps/web/Dockerfile` is removed.
- `docker-compose.yml` replaces the `web` service with `applicant` and `admin` services.
- Root `package.json` gains per-app `dev:*`/`build`/`lint` scripts; `typecheck` runs the packages
  project plus each app project; `tsconfig.json` drops the ambiguous root `@/*` alias and adds
  `@talentos/ui`; `.env.example` replaces `WEB_PORT` with `APPLICANT_PORT`/`ADMIN_PORT` and adds the
  cross-link URLs.

## Deployment Commands

```powershell
npm.cmd install
npm.cmd run db:generate
docker compose up postgres -d
npm.cmd run db:migrate
npm.cmd run db:seed
docker compose up applicant admin -d --build
```

## Validation URLs

Applicant container (`http://localhost:3100`): `/`, `/apply`, `/signup`, `/login`, `/2fa/setup`,
`/application`, `POST /api/applications`, `POST /api/ai/mentor`.

Admin container (`http://localhost:3200`): `/`, `/applications`, `/applications/[id]`, `/programs`,
`/settings`.

Isolation: admin routes return 404 on the applicant container and applicant routes return 404 on the
admin container.

## Assumptions

- Local `.env` overrides default ports and is not committed.
- Isolation is by separate host ports in this iteration; reverse-proxy/subdomain routing is future work.
