# v0.18.1 Plan Test Scenarios Requirement

Code version: `v0.18.1`

Previous baseline: `v0.18.0`

Plan status: Implemented

## Summary

Governance-only patch. Auditing `v0.17.0`–`v0.18.0` found that the Engineering Journal plan
(`docs/plans/v0.17.0_Engineering_Journal_MVP.md`) never named scenario-level test cases — only a
"Verification: see test-results doc" link — so the feature shipped with strong unit coverage
(`journal.test.ts`, 23 tests) and zero scenario-level regression coverage, a gap only caught later by
manual audit. This patch makes "plan the test scenarios up front" a structural requirement instead of
an easy-to-skip line of prose.

## Scope

- Add `docs/plans/TEMPLATE.md`, required for every new plan, with a mandatory Test Scenarios section
  (actor, preconditions, steps, expected result, explicit automation call).
- Add `docs/testing/TEMPLATE.md`, required for every new test-results doc, with a Scenario Results
  table mapping 1:1 to the plan's Test Scenarios.
- Add `docs/sdlc.md` Version and Documentation Control rule 7 pointing at both templates and requiring
  every plan scenario to land in `docs/Regression_Scenarios.md` (automated or an explicit Known Gap)
  in the same iteration.
- Update `CONTRIBUTING.md` and `.github/pull_request_template.md` so the requirement is visible at
  both write-time (quickstart) and review-time (PR checklist).
- Add `D-076` to `docs/Decision_Log.md` and this plan/test-results pair, itself written against the
  new template.

## Out of Scope

- Retroactively rewriting already-shipped plans (`v0.17.0`, `v0.18.0`, etc.) to add Test Scenarios
  sections after the fact — the existing scenario gaps for those versions are already tracked in
  `docs/Regression_Scenarios.md` Known Gaps (added during the `v0.18.0` compliance pass) rather than
  retrofitted here.
- Automated tooling that blocks a PR if the Test Scenarios section is empty/boilerplate — this patch
  makes it a required, reviewer-checked item, not a CI gate. A future iteration could add a CI check
  (e.g. grep the plan for unfilled template placeholders) if manual review proves insufficient.
- No product code, schema, Docker configuration or package file is touched.

## Test Scenarios

This is a process/documentation change, not an application feature — there is no user-facing behavior
to exercise through `scripts/regression/run.ts`. The "scenarios" below are the acceptance criteria for
the process itself, each with a stated verification method instead of a runtime regression scenario.

- **Scenario**: A new plan doc is written using the required template.
  **Actor**: any future contributor (human or AI) starting a new versioned change.
  **Preconditions**: `docs/plans/TEMPLATE.md` exists and `docs/sdlc.md` rule 7 references it.
  **Steps**: copy `docs/plans/TEMPLATE.md` to `docs/plans/vX.Y.Z_<Title>.md` and fill it in.
  **Expected result**: the resulting plan has a non-empty, non-boilerplate Test Scenarios section
  before implementation is considered complete.
  **Automation**: Deferred — this is verified by PR review against the new checklist item (see
  `.github/pull_request_template.md`), not by `scripts/regression/run.ts`. Recorded here rather than
  in `docs/Regression_Scenarios.md`, which covers application-behavior scenarios, not documentation
  process checks.

- **Scenario**: This same patch's own plan/test-results pair follows the new template.
  **Actor**: this change's author.
  **Preconditions**: none.
  **Steps**: write `docs/plans/v0.18.1_Plan_Test_Scenarios_Requirement.md` and
  `docs/testing/v0.18.1_Plan_Test_Scenarios_Requirement_Test_Results.md` using the templates just
  added.
  **Expected result**: both documents match the template structure (verified directly — this plan and
  its test-results doc are the evidence).
  **Automation**: Added this iteration, as a direct self-check rather than a `regression:*` npm
  script — see Verification below.
