# v0.10.1 — Keycloak OTP Policy Fix — Implementation Plan

Code version: `v0.10.1`

## Objective

Fix an internal server error that occurred when any user completed first-login authenticator-app (TOTP)
setup. This blocked the super admin (and every new user with a `CONFIGURE_TOTP` required action) from
signing in.

## Root cause

The realm import (`keycloak/import/talentos-realm.json`) set `otpPolicyType: "totp"` but omitted the
rest of the OTP policy, so Keycloak used `otpPolicyPeriod = 0`. During TOTP validation,
`TimeBasedOTP$Clock.getCurrentInterval` computes `time / period`, raising
`java.lang.ArithmeticException: / by zero` and returning Keycloak's "internal server error" page after
the 2FA form was submitted.

## Fix

Add the full OTP policy to the realm import:

```json
"otpPolicyType": "totp",
"otpPolicyAlgorithm": "HmacSHA1",
"otpPolicyDigits": 6,
"otpPolicyPeriod": 30,
"otpPolicyInitialCounter": 0,
"otpPolicyLookAheadWindow": 1,
```

For the **already-running** Keycloak (realm persisted in the `keycloak-postgres` volume), the same
policy was applied live and non-destructively via `kcadm.sh`:

```
kcadm.sh config credentials --server http://localhost:8080 --realm master --user admin --password admin
kcadm.sh update realms/talentos -s otpPolicyType=totp -s otpPolicyAlgorithm=HmacSHA1 \
  -s otpPolicyDigits=6 -s otpPolicyPeriod=30 -s otpPolicyInitialCounter=0 -s otpPolicyLookAheadWindow=1
```

## Regression guard

New unit test `packages/auth-web/src/realm-otp.test.ts` reads the realm import and asserts
`otpPolicyType === "totp"` with a non-zero `otpPolicyPeriod` and `otpPolicyDigits`, so this
misconfiguration cannot silently return in a fresh environment. Regression suite: 46 → 47 tests.
