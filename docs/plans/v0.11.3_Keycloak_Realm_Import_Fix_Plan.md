# v0.11.3 — Keycloak Realm-Import Fix — Implementation Plan

Code version: `v0.11.3`

Baseline commit: `2fd6fce`

Previous baseline: `v0.11.2`

## Problem

The `talentos-provisioner` service-account client added in `v0.11.0` (D-053) was written into the realm
import file `keycloak/import/talentos-realm.json` using an **invalid field, `serviceAccountClientRoles`**,
on the client's `ClientRepresentation`. Keycloak's import parser does not recognize this field and
aborts at the JSON-parse step — **before** the realm-exists check — so `start-dev --import-realm` fails
on **every** startup and Keycloak crash-loops. This breaks any fresh deployment (and every restart of an
existing one) at the identity layer: no portal can authenticate.

The defect escaped v0.11.0 testing because that iteration verified the provisioner only via a **live
`kcadm.sh` patch** on an already-running realm; the baked-in realm-import path was never exercised
against a clean Keycloak volume. This was found during a v0.11.2 end-to-end run.

## Fix

In `keycloak/import/talentos-realm.json`:

1. **Remove** the invalid `serviceAccountClientRoles` object from the `talentos-provisioner` client.
2. **Add** a `service-account-talentos-provisioner` user to the realm `users` array — the canonical
   Keycloak realm-import representation of a service account's role mappings:

   ```json
   {
     "username": "service-account-talentos-provisioner",
     "enabled": true,
     "serviceAccountClientId": "talentos-provisioner",
     "clientRoles": { "realm-management": ["manage-users", "view-realm", "query-users"] }
   }
   ```

No application code, schema, or data-model change. The client itself (confidential,
`serviceAccountsEnabled`, secret) is unchanged; only how its realm-management roles are expressed in the
import.

## Verification

Destructive fresh-import test (see the test-results doc): wipe the `keycloak-postgres` volume and let
`--import-realm` run the corrected file from scratch. Confirm Keycloak starts, the realm imports without
error, the provisioner authenticates via `client_credentials`, and the Keycloak Admin `/users` API
returns 200 (proving `manage-users`/`query-users` are mapped). Re-run the portal smoke suite.

## Deployment

- **Fresh environment:** the corrected import is picked up automatically (the import dir is a read-only
  volume mount; no image rebuild needed) — `docker compose up` and the realm imports cleanly.
- **Already-running environment** whose realm was previously patched live via `kcadm.sh`: no action
  required (the service account already exists); the fix simply makes the on-disk import valid so future
  restarts/redeploys no longer crash.
