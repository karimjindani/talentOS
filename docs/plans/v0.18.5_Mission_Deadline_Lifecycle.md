# v0.18.5 Mission Deadline & Lifecycle

Code version: `v0.18.5`

Previous baseline: `v0.18.4`

Plan status: Implemented

## Summary

`v0.18.0` gave each accepted applicant an assigned mission per week, but the assignment had no
concept of time: an applicant could sit on a `NOT_STARTED`/active assignment indefinitely with no
deadline, and a rejected submission always reset the applicant back to a fresh Week 1 attempt
regardless of which week actually failed. This version gives every mission assignment an explicit
lifecycle — accept, deadline, grace period, late submission, auto-advance to the next week on
acceptance, and reassignment on rejection — enforced by an idempotent external sweep rather than
by request-time checks, so a deadline is enforced even if no applicant or admin ever loads the app.

## Scope

- `Mission.deadlineHours` (default 168h / 7 days) and `Mission.gracePeriodHours` (default 24h),
  configurable per mission.
- `MissionAssignment.acceptedAt` / `deadlineAt` / `graceEndsAt`, computed from the mission's own
  hours at the moment of explicit acceptance — an assignment that is never accepted never expires.
- `MissionAssignmentStatus` becomes `NOT_STARTED` (new default) `→ ACCEPTED → IN_PROGRESS →
  PENDING_EVALUATION | LATE_SUBMITTED`, with `OVERDUE` and terminal `FAILED` as deadline-driven
  side states, and `PASSED` / `REPEAT` as review outcomes. Replaces the `v0.18.0` two-state
  `ACTIVE`/`SUBMITTED` model.
- An explicit **Accept Mission** action (`acceptMissionAssignment`) — the applicant must accept a
  `NOT_STARTED` assignment before evidence can be drafted; accepting is what starts the deadline
  clock.
- A standalone, idempotent **deadline sweep** (`packages/db/src/mission-deadlines.ts`,
  `scripts/mission-deadlines/sweep.ts`, `npm run mission-deadlines:sweep`) run as an external
  scheduled job (cron), not as part of any request path, per explicit product direction to keep
  scheduled background work out of the app process for future scaling. Two idempotent,
  status-scoped phases: `ACCEPTED|IN_PROGRESS` past `deadlineAt` → `OVERDUE`; `OVERDUE` past
  `graceEndsAt` with no submission → `FAILED` + `Application.status = DISQUALIFIED`. Re-running the
  sweep any number of times cannot double-transition or double-notify, because each phase's update
  is scoped to the source status.
- A submission made after `deadlineAt` but before `graceEndsAt` is still accepted, recorded as
  `LATE_SUBMITTED` instead of rejected outright.
- Accepting a submission auto-advances the applicant to the next week's mission
  (`assignWeekMissionToAcceptedApplicantTx(weekNumber + 1)`), capped at `FINAL_PROGRAM_WEEK = 4` —
  accepting the Week 4 submission does not create a Week 5 assignment.
- A `REPEAT` review decision reassigns the applicant a different published mission (this version:
  back at Week 1); the failed mission is never reused for the reassignment.
- New `ApplicationStatus` values `DISQUALIFIED` (grace period expired with no submission — terminal
  for this version; a future admin-driven "rejoin from Week 1" path is explicitly out of scope) and
  `AWAITING_MISSION_ASSIGNMENT` (no alternate mission was available to reassign — every
  `ORG_ADMIN`/`TECH_LEAD` in the tenant is notified to assign one manually).
- Migration `20260714090000_mission_deadlines_and_lifecycle`.

## Out of Scope

- Letting a disqualified (missed-deadline) applicant rejoin the program from Week 1 — deferred
  pending a Back Office admin workflow decision.
- Reassigning the specific mission a `REPEAT` decision failed — the applicant only ever gets a
  different published mission, never the one they were just rejected on.
- Any UI surfacing of the deadline/grace countdown or the accept action beyond the minimum needed
  to exercise the state machine (dashboard/tasks/program wiring is a later iteration — see
  `docs/plans/v0.19.1_Dashboard_Wiring_And_Same_Week_Repeat.md`).
- Notifying the applicant themselves when they are marked `FAILED`/`DISQUALIFIED` by the sweep (only
  reviewers are notified on the no-alternate-mission path in this version).

## Test Scenarios

- **Scenario**: Explicit accept starts the deadline/grace clock.
  **Actor**: accepted applicant with a `NOT_STARTED` mission assignment.
  **Preconditions**: applicant has been assigned a published mission for their current week and has
  not yet accepted it.
  **Steps**: applicant calls the accept action against their assignment.
  **Expected result**: assignment status becomes `ACCEPTED`; `acceptedAt` is set to now;
  `deadlineAt` = `acceptedAt` + the mission's `deadlineHours`; `graceEndsAt` = `deadlineAt` + the
  mission's `gracePeriodHours`; accepting twice (already-`ACCEPTED`) is rejected.
  **Automation**: Deferred — covered at unit level only
  (`packages/db/src/mission-assignments.test.ts`); no `scripts/regression/run.ts` scenario drives
  the accept action through a real route yet.

- **Scenario**: Idempotent deadline sweep marks a stale assignment `OVERDUE`.
  **Actor**: system (external cron running the sweep script).
  **Preconditions**: an `ACCEPTED`/`IN_PROGRESS` assignment whose `deadlineAt` has passed but
  `graceEndsAt` has not.
  **Steps**: run `sweepMissionDeadlines(now)` twice in a row.
  **Expected result**: the assignment transitions to `OVERDUE` on the first run; the second run is
  a no-op (no duplicate transition, no duplicate notification) because the update is scoped to the
  source status.
  **Automation**: Deferred — covered at unit level only (`packages/db/src/mission-deadlines.test.ts`);
  no scenario-level automation of the external sweep script itself.

- **Scenario**: Grace-period expiry is terminal (`FAILED` + `DISQUALIFIED`).
  **Actor**: system (sweep).
  **Preconditions**: an `OVERDUE` assignment whose `graceEndsAt` has passed with no submission.
  **Steps**: run the sweep.
  **Expected result**: assignment → `FAILED`; `Application.status` → `DISQUALIFIED`; no rejoin path
  exists yet (explicit product decision — deferred).
  **Automation**: Deferred — covered at unit level only (`packages/db/src/mission-deadlines.test.ts`).

- **Scenario**: A late submission inside the grace period is still accepted.
  **Actor**: accepted applicant whose assignment is `OVERDUE` (deadline passed, grace has not).
  **Preconditions**: assignment `OVERDUE`.
  **Steps**: applicant submits evidence.
  **Expected result**: submission is accepted and recorded as `LATE_SUBMITTED`, not rejected.
  **Automation**: Deferred — covered at unit level only (`packages/db/src/submissions.test.ts`).

- **Scenario**: Accepting a submission auto-advances the applicant, capped at the final week.
  **Actor**: ORG_ADMIN or TECH_LEAD reviewer.
  **Preconditions**: applicant's Week `N` submission is `PENDING_EVALUATION`.
  **Steps**: reviewer accepts the submission.
  **Expected result**: a new `NOT_STARTED` assignment for Week `N + 1` is created automatically
  (`N < FINAL_PROGRAM_WEEK`); accepting the Week 4 submission creates no Week 5 assignment.
  **Automation**: Deferred — covered at unit level only (`packages/db/src/submissions.test.ts`).

- **Scenario**: Rejection reassigns an alternate mission; no alternate notifies reviewers.
  **Actor**: ORG_ADMIN or TECH_LEAD reviewer.
  **Preconditions**: applicant's submission is under review; a `REPEAT` decision is made.
  **Steps**: reviewer rejects with a `REPEAT` decision, once when an alternate published mission
  exists for the target week, once when it doesn't.
  **Expected result**: with an alternate available, a new assignment is created for a different
  published mission (never the failed one); with no alternate, no assignment is created,
  `Application.status` becomes `AWAITING_MISSION_ASSIGNMENT`, and every `ORG_ADMIN`/`TECH_LEAD` in
  the tenant receives a `WARNING` notification — the failed mission is never reassigned and the
  applicant is never removed.
  **Automation**: Deferred — covered at unit level only (`packages/db/src/mission-assignments.test.ts`).

- **Scenario (negative/edge)**: A `FAILED` assignment cannot receive new evidence.
  **Actor**: a disqualified applicant.
  **Preconditions**: assignment status `FAILED` (grace period expired).
  **Steps**: applicant attempts to submit evidence against the failed assignment.
  **Expected result**: the submission is rejected — `FAILED` is a terminal state for the assignment.
  **Automation**: Deferred — covered at unit level only (`packages/db/src/submissions.test.ts`).

All scenarios above are recorded in `docs/Regression_Scenarios.md` Known Gaps rather than silently
missing: they have solid unit coverage but no dedicated `scripts/regression/run.ts` scenario drives
the accept/sweep/reject-reassignment flow through the actual applicant/admin actions end-to-end yet.

## Security and Data Rules

- The sweep is a standalone script with its own Prisma client import — it must never be reachable
  from an unauthenticated HTTP route.
- `acceptMissionAssignment` is scoped by `{ id, tenantId, applicantId }` — an applicant can only
  accept their own assignment.
- The no-alternate-mission notification path never reassigns the failed mission and never removes
  the applicant from the tenant/program.
- `DISQUALIFIED` and `AWAITING_MISSION_ASSIGNMENT` are terminal `ApplicationStatus` values with no
  admin-initiated transition in `packages/auth/src/workflow.ts` — only the sweep/reassignment logic
  sets them.
