# v0.10.3 — Tenant Isolation Fix (D-051)

Code version: `v0.10.3` · Previous: `v0.10.2` · Date: `2026-07-02`

## Problem

Admin authorization was decided from two unlinked inputs: the role came from the **realm-wide** Keycloak
access token (`packages/auth-web/src/auth.ts` → `orgRole`/`platformRole`) while the tenant came from the
**Host header** (`packages/auth/src/tenant.ts`). `can()` checked only the role. An
`ORG_ADMIN`/`HR`/`TECH_LEAD` of tenant A could therefore open `tenantB.<host>` and operate on tenant B —
program CRUD, application accept/reject, branding/logo overwrite, and **candidate-CV download** were all
reachable. `assertTenantScopedAccess(resource.tenantId, resolvedTenant.id)` compared B-vs-B and passed;
it never checked the actor. The correct data (`TenantMembership` per admin) already existed in the DB but
was never consulted at request time. (Accepted as a known limitation under D-048.)

## Fix — bind authorization to `TenantMembership`

Two-layer model: the Keycloak realm role remains the **coarse portal-entry** gate (middleware); the DB
`TenantMembership` becomes the **authoritative per-tenant** gate. SUPER_ADMIN bypasses.

- **New primitives**
  - `getActorTenantRoles(email, tenantId)` — `packages/db/src/users.ts`; case-insensitive email match →
    `{ userId, roles: TenantRole[] } | null`.
  - `tenantRolesGrant(capability, roles)` — `packages/auth/src/permissions.ts`.
- **Shared guard** — `apps/admin/lib/tenant-guard.ts`: `resolveTenantAccess(capability?)` (discriminated
  result, for route handlers + layout) and `requireTenantAccess(capability?)` (redirects, for server
  components/actions). Resolves session → host tenant → membership; SUPER_ADMIN bypass.
- **Wiring**
  - Admin layout (`apps/admin/app/layout.tsx`) renders an inline "access denied" notice for non-members
    instead of children (no redirect → no `/forbidden` loop) — gates every page read at once.
  - Actions: `requireProgramManager` (`programs/actions.ts`), `reviewApplication`
    (`applications/[id]/page.tsx`), `saveTenantBranding` (`settings/actions.ts`).
  - Route handlers: CV download (`api/files/[id]/download/route.ts`, previously only checked for *any*
    authenticated session) and operations health (`api/operations/health/route.ts`).
- **Defense-in-depth** — `updateProgram`/`setProgramStatus` (`packages/db/src/programs.ts`) and
  `applyStatusTransition` (`packages/db/src/applications.ts`) now write via
  `updateMany({ where: { id, tenantId } })` and throw if `count === 0`.

## Scope

Code-only; **no schema/migration**, no data-model change. Out of scope (tracked separately): persist
`keycloakSubjectId` on login for all roles + full email normalization + `email_verified` gate;
Keycloak user/role auto-provisioning (`v0.3.1`); reserved-slug blocklist; `Application` unique constraint.

See test results: `docs/testing/v0.10.3_Tenant_Isolation_Fix_Test_Results.md`. Decision: `D-051`.
