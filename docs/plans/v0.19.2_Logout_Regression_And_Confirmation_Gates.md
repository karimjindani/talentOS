# v0.19.2 Logout Regression Fix & Confirmation Gates

Code version: `v0.19.2`

Previous baseline: `v0.19.1`

Plan status: Implemented

## Summary

Two small, unrelated fixes that predate the `v0.18.5`–`v0.19.1` mission-lifecycle work but were
never versioned or committed. (1) The `v0.14.3`/D-066 applicant dashboard Logout button had gone
missing — a regression introduced when the `feat/applicant-ai-mentor-skeleton` branch merged
(PR #45) and reverted part of an earlier main-branch merge — and is restored. (2) `AGENTS.md`
gains a `Confirmation Gates` section requiring explicit user confirmation before starting a
versioned documentation-update process or pushing to a remote branch, closing a gap where the
repo's process docs described *what* to do for versioning/pushing but never said to check in with
the user first.

## Scope

- Restore the sidebar Logout button in `ApplicantShell.tsx` (`<form action={logoutAction}>`,
  mirroring the existing admin-portal logout pattern) — the dashboard replaces `PortalHeader`
  entirely, so the shell must carry its own sign-out affordance.
- Fix a `vitest.config.ts` alias gap: `apps/applicant/tsconfig.json`'s `"@/*" -> "./*"` path was not
  mirrored in the Vitest resolver, so any test importing through `@/...` (as
  `ApplicantShell.test.ts` now does via the logout action) failed to resolve. Adds a scoped
  `@/(.+)` → `apps/applicant/$1` alias.
- `ApplicantShell.test.ts` mocks `@/lib/logout-action` the same way
  `apps/applicant/lib/logout-action.test.ts` already does, since the real module chain
  (`@/auth` → `next-auth` → `next/server`) needs the Next.js runtime and isn't resolvable under
  plain Vitest.
- `AGENTS.md` **Confirmation Gates**: stop and ask before (1) starting the documentation-update
  process for a versioned iteration, and (2) pushing commits to a remote branch (including
  force-pushes and fast-forward pushes alike).

## Out of Scope

- Any other governance change to `AGENTS.md`.
- Re-auditing the rest of the `feat/applicant-ai-mentor-skeleton` merge for other reverted
  behavior — this patch only restores the one regression that was reported (the missing Logout
  button).

## Test Scenarios

- **Scenario**: Accepted applicant can sign out from the dashboard.
  **Actor**: accepted applicant on `/dashboard` (or any dashboard sub-page).
  **Preconditions**: applicant is signed in and viewing the dashboard shell.
  **Steps**: applicant clicks Logout in the sidebar footer.
  **Expected result**: the existing `logoutAction` server action runs (OIDC RP-initiated logout,
  `v0.10.2`/`v0.14.3`), ending the session the same way the admin portal's logout does.
  **Automation**: Already automated — `apps/applicant/components/ApplicantShell.test.ts` (13 tests)
  asserts the shell renders with the mocked logout action wired in; `apps/applicant/lib/
  logout-action.test.ts` already covers the action itself. No new scenario-level regression was
  added; this is a UI-restoration fix over already-tested behavior.

- **Scenario (negative/edge)**: A test importing through the `@/` alias resolves under Vitest.
  **Actor**: any test file under `apps/applicant` using `@/...` imports.
  **Preconditions**: none.
  **Steps**: run `npm test`.
  **Expected result**: `ApplicantShell.test.ts` (and any future `@/`-importing applicant test)
  resolves via the new scoped alias instead of failing to find the module.
  **Automation**: Already automated — exercised by every `npm test` run; no dedicated test needed
  since a resolution failure would surface as every test in the file failing to even collect.

- **Scenario**: The documentation-update process pauses for confirmation.
  **Actor**: any agent (Claude or otherwise) working in this repo.
  **Preconditions**: none.
  **Steps**: an agent is asked to do work that would start a versioned documentation update, or to
  push commits to a remote branch.
  **Expected result**: the agent stops and asks for explicit confirmation before proceeding, per
  the new `AGENTS.md` section.
  **Automation**: Deferred — this is a process/governance rule for human-directed agent behavior,
  not testable by an automated suite. Verified by direct observation this session (the docs-update
  process for `v0.18.5`–`v0.19.1` did in fact pause for confirmation, and pushing has not been
  attempted without asking).

## Security and Data Rules

None — no new access-control surface. The logout action reuses the existing OIDC RP-initiated
logout flow (`buildTenantLogoutUrl`, `packages/auth-web`) unchanged.
