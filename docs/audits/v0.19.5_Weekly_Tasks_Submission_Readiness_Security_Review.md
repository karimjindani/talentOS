# v0.19.5 Weekly Tasks And Submission Readiness Security Review

Version: `v0.19.5`

Review date: 2026-07-19

Status: Implemented controls reviewed; residual risks recorded

Scope: Weekly tasks/resources, applicant task completion, Engineering Journal readiness,
submission evidence reachability, final submission state changes and Admin content management.

## Trust Boundaries

- Browser input crosses into Applicant or Admin server actions.
- Tenant, applicant, assignment and reviewer identity comes from authenticated server context, not form
  fields.
- PostgreSQL stores task completion, journal, submission and assignment state.
- DNS and remote HTTP(S) services are untrusted external dependencies.
- Markdown task content is admin-authored but still rendered as untrusted display content.

## Threat And Risk Register

| ID | Threat | Impact | Implemented mitigation | Residual risk | Status |
| --- | --- | --- | --- | --- | --- |
| SR-1 | Cross-tenant task, journal, submission or content access | Disclosure or mutation of another tenant's data | DB helpers require `tenantId`; applicant writes also bind user/program/week/assignment; Admin actions retain tenant guard and existing capability checks. | A future helper could omit a scope predicate; code review and regression coverage remain necessary. | Mitigated |
| SR-2 | Task-completion leakage between applicants, programs or weeks | False readiness | Completion reads filter tenant/user and task relation program/week; writes validate the active accepted-program week; unique key is tenant/user/task. | Completion intentionally survives same-week repeat, so product changes to task semantics would need a migration/design review. | Mitigated |
| SR-3 | Journals from an old or different attempt count toward readiness | Premature submission or wrong records locked | Readiness and locking filter exact `missionAssignmentId`; future dates are excluded; previous-attempt review queries are separately scoped. | Legacy unlinked journals do not count, which may require an explicit backfill decision for old data. | Mitigated |
| SR-4 | Direct SSRF to localhost, metadata or private/reserved networks | Internal service/data exposure | Only HTTP(S); credentials rejected; host/IP blocklists cover loopback, link-local, private, reserved and metadata destinations for IPv4/IPv6. | Public services can still expose sensitive content by design; allowlists are limited to GitHub/Loom where business rules require them. | Mitigated |
| SR-5 | Redirect from a public URL into a private network | SSRF after initial validation | Every redirect is parsed, DNS-resolved and checked again; redirect count is bounded. | Redirect/DNS behavior remains an external dependency and must be retested when the HTTP client changes. | Mitigated |
| SR-6 | DNS rebinding between validation and request | Request reaches an address that was not validated | Every resolved address is screened and the request lookup is pinned to a validated address; both Node single-address and `all` callback modes are tested. | Resolver and remote routing can still fail; failures block submission rather than being treated as reachable. | Mitigated |
| SR-7 | External timeout, outage or GitHub rate limiting | Valid applicant evidence is temporarily blocked | Bounded timeout, HEAD with bounded GET fallback and field-specific errors; no state mutation on failure. | Fail-closed behavior can cause temporary false negatives. There is no asynchronous retry/cache or reviewer override in this iteration. | Accepted |
| SR-8 | Concurrent or repeated final submissions | Duplicate transition, notifications or journal locks | Readiness/evidence are rechecked in a short transaction; status-scoped `updateMany` permits one winner; journal lock update is attempt scoped. | Database/network failures after commit may still require operational reconciliation of downstream effects. | Mitigated |
| SR-9 | Unsafe Markdown executes script or raw HTML | Applicant-side XSS | `SafeMarkdown` renders a constrained structure and leaves HTML as text; raw HTML is not enabled. | The custom renderer supports a limited Markdown subset and must be reviewed before adding new constructs. | Mitigated |
| SR-10 | Pending YouTube resource is mistaken for approved content | Misleading or broken learning path | Null URL is an explicit pending state in Applicant and Admin UI; no placeholder external URL is followed. | The actual introduction video and final URL are still operational content dependencies. | Accepted |
| SR-11 | Malicious semicolon deployment input bypasses per-link checks | One unsafe URL hidden inside a combined string | Central parser splits, trims, removes blanks/duplicates, caps at ten and validates/checks every URL independently. | Existing malformed historical strings may fail new readiness until corrected. | Mitigated |
| SR-12 | Network calls held inside a DB transaction | Connection starvation and lock contention | Reachability runs before the transaction; only critical readiness/evidence/status checks and writes occur inside. | State can change between preflight and transaction, so evidence and readiness are rechecked and mismatches fail closed. | Mitigated |

## Authorization Review

- No Keycloak roles, permission-matrix entries or authentication flows changed.
- Applicant completion/journal/submission operations use server-resolved applicant and tenant identity.
- Admin weekly-content operations reuse `manageProgramContent` and tenant ownership checks.
- Submission review roles and status transitions are unchanged.
- The Ops Console continues to display regression output under its existing Keycloak guard.

## Data Integrity Review

- `UserTaskCompletion` is unique on `[tenantId, userId, taskId]` and uses an idempotent write path.
- Existing one-entry-per-applicant/date uniqueness remains in force for journal rows.
- Four-journal readiness is `>= 4`, exact-attempt scoped and excludes future entry dates.
- Submission failure occurs before journal locks/status changes; transaction guards defend the final
  transition and concurrent callers.
- `Submission.deploymentUrl` remains a normalized semicolon-separated string for compatibility; every
  read path must use the central parser before display or validation.
