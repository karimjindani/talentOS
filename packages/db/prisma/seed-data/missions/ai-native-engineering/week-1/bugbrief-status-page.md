# Build a Public Incident Status Page

## Objective

Turn an ambiguous reliability communication need into a deployed public status page.

## Mission Brief

A small SaaS team has started receiving support messages whenever their app slows down. They need a public status page called BugBrief so users can quickly see whether the service is healthy, degraded, or under maintenance.

Business context:
The client does not need a full backend yet. They need a professional static page that communicates trust, current service state, recent incidents, and a clear support contact path.

The page should help visitors understand:
- Current platform status
- What incidents or maintenance windows mean
- Where to get support
- How transparent the company is about reliability

You are expected to use AI tools, but you must document how AI helped you and what decisions you made independently.

SEM lifecycle:
1. Discover: write at least 5 clarification questions about reliability communication and customer expectations.
2. Analyze: summarize the users, pain points, and trust risks.
3. Specify: create a short PRD with page goals, user stories, out-of-scope items, and acceptance criteria.
4. Design: create a wireframe or page structure for the status page.
5. Build: build a responsive status page with realistic status and incident sections.
6. Test: create and execute a manual test checklist.
7. Deploy: publish the site using Vercel, Netlify, GitHub Pages, or similar.
8. Present: record a 3-5 minute Loom demo.
9. Reflect: write an engineering journal entry.
10. Production Readiness Review: decide whether the page is ready for real customer traffic.

## Deliverables

- PRD
- User stories
- Acceptance criteria
- Wireframe or page structure
- GitHub repository URL
- Deployment URL
- README
- Manual test checklist
- Loom demo video
- Engineering journal entry
- AI usage disclosure

## Acceptance Criteria

- The status page is publicly accessible.
- The page is responsive on desktop and mobile.
- The page shows at least one current status, one incident history item, and one support/contact path.
- The GitHub repository contains a clear README.
- The PRD and user stories are included in documentation.
- The deployment URL works without login.
- AI usage is documented transparently.
- Manual testing evidence is included.

## Evaluation Criteria

Bronze: working deployed status page with README and basic demo.
Silver: includes PRD, user stories, acceptance criteria, and test checklist.
Gold: clear incident communication, responsive polished UI, and transparent AI usage disclosure.
Platinum: professional reliability communication, accessible UI, strong trust-building copy, and thoughtful production readiness reflection.
